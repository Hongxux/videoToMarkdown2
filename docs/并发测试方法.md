? 最优做法不是“直接把并发拉满”，而是用容量拐点测试（Knee Test）+ 长稳态验证（Soak）。 

  为什么这是该领域最佳

  - 你这类链路是“Java编排 + Python推理 + 外部API限流 + 本地FFmpeg/磁盘”的混合系统，并 
    发瓶颈是多源的，不是单点。
  - 你项目里已有硬约束：Vision API 60 req/min（docs/architecture/upgrade-
    log.md:694）、gpu_max_workers: 1（config/video_config.yaml:96）、
    transcription_concurrency: 1（config/video_config.yaml:230）、多处
    parallel_hard_cap（config/module2_config.yaml:347、config/
    module2_config.yaml:375）。

  推荐测试流程（实战）

  - 定义SLO：例如 p95总时延、成功率、429率、队列等待时间。
  - 做分层压测：入口REST并发、Java->Python gRPC并发、外部API调用并发分别测，再做端到  
    端。
  - 做阶梯升压：并发按 1→2→4→6→8→...，每档跑10–15分钟。
  - 找拐点：当并发继续上升但吞吐增幅 <5% 且 p95/错误率/429明显恶化，即到达容量拐点。  
  - 运营并发取值：取拐点的 70%~80% 作为生产安全并发。
  - 最后做Soak：在该并发下连续跑2–8小时，确认无内存泄漏、队列堆积、事件循环异常。     

  核心判据（比“QPS最高”更重要）
                                                                                      
  - 任务吞吐稳定增长、p95可控、429和超时可控、资源曲线平稳（CPU/GPU/内存/磁盘IO）。   
  - 用 Little 定律估算：并发 ≈ 到达率 × 平均任务时长，再与各硬上限取最小值。
