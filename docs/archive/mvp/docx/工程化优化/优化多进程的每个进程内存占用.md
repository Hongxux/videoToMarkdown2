## 一、核心优化：OpenCV（150-200MB/进程）—— 解决多进程重复加载
OpenCV是内存占用最大的模块，且「每进程独立加载」是核心浪费点，优先解决这个问题能立竿见影。

### 1. 多进程模式优化（Linux/macOS首选）
Python多进程的`fork`模式会继承父进程的内存空间，OpenCV只需在父进程加载一次，所有子进程复用，避免重复加载：
```python
import multiprocessing as mp
import cv2
import psutil
import os

def opencv_worker(frame_data):
    """子进程直接复用父进程加载的OpenCV，无需重新加载"""
    gray = cv2.cvtColor(frame_data, cv2.COLOR_BGR2GRAY)
    return gray.shape

if __name__ == "__main__":
    # 强制使用fork模式（Linux/macOS默认，Windows不支持）
    mp.set_start_method('fork')
    
    # 父进程先加载OpenCV（仅加载1次）
    parent_pid = os.getpid()
    parent_mem = psutil.Process(parent_pid).memory_info().rss / 1024 / 1024
    print(f"父进程(PID:{parent_pid})加载OpenCV后内存: {parent_mem:.2f}MB")
    
    # 创建进程池（复用进程，避免频繁创建销毁导致重复加载）
    pool = mp.Pool(processes=4)  # 限制进程数，避免内存叠加
    
    # 模拟帧处理
    test_frame = cv2.imread("test.jpg")  # 替换为你的帧路径
    results = pool.map(opencv_worker, [test_frame]*10)
    
    pool.close()
    pool.join()
    
    # 验证：子进程内存不会因OpenCV重复加载暴涨
    child_mem = psutil.Process(os.getpid()).memory_info().rss / 1024 / 1024
    print(f"处理完成后内存: {child_mem:.2f}MB（无暴涨）")
```
**效果**：4个进程的OpenCV内存占用从 `4*200MB=800MB` 降至 `200MB`，节省75%。

### 2. Windows兼容方案（Spawn模式）
Windows不支持`fork`，改用「共享内存传递帧数据」，子进程不重新加载OpenCV核心库：
```python
import multiprocessing as mp
import cv2
import numpy as np

def worker(shared_frame, frame_shape):
    """子进程从共享内存读取帧，复用父进程的OpenCV实例"""
    # 将共享内存转为numpy数组（无新内存分配，无需重新加载OpenCV）
    frame = np.frombuffer(shared_frame.get_obj(), dtype=np.uint8).reshape(frame_shape)
    gray = cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)
    return gray.mean()

if __name__ == "__main__":
    # 父进程加载帧并创建共享内存
    frame = cv2.imread("test.jpg")
    frame_shape = frame.shape
    frame_size = frame.nbytes  # 帧总字节数
    
    # 创建共享内存（子进程直接读取，无需拷贝）
    shared_frame = mp.Array('B', frame_size, lock=False)
    # 将帧数据写入共享内存
    np.frombuffer(shared_frame.get_obj(), dtype=np.uint8)[:] = frame.flatten()
    
    # 启动子进程（不重新加载OpenCV）
    p = mp.Process(target=worker, args=(shared_frame, frame_shape))
    p.start()
    p.join()
```

### 3. 精简OpenCV运行时内存
禁用不必要的功能，减少单进程内OpenCV的内存占用：
```bash
# 运行时功能裁剪（主链路默认启用）
OPENCV_OPENCL_RUNTIME=disabled
CV_DISABLE_OPENCV_OPT=0   # 需要时可设为 1
```
- 启动阶段设置 `OPENCV_OPENCL_RUNTIME=disabled`，避免 OpenCL 运行时与额外组件加载。
- Worker 初始化调用 `cv2.ocl.setUseOpenCL(False)`，防止 OpenCL 相关内存常驻。
- 可选开关：`CV_DISABLE_OPENCV_OPT=1` 时执行 `cv2.setUseOptimized(False)`，在内存敏感场景进一步收敛占用（以少量性能换内存）。

## 二、NumPy（50-100MB）：紧凑存储+减少拷贝
### 1. 改用更紧凑的数据类型（核心优化）
根据业务场景降低数据精度，内存可直接减半：根据我们的判断需求，可以采用如下方案
工程落地：主链路 CV 中间计算默认使用 float32，可通过环境变量 `CV_FLOAT_DTYPE=64` 回退精度。
```python
import numpy as np
import sys

# 优化前：float64（8字节/元素）→ 占用~80MB（1000万元素）
arr64 = np.random.rand(10000000)
print(f"float64内存: {sys.getsizeof(arr64) + arr64.nbytes:.2f}MB")  # ~76.3MB

# 优化后：float32（4字节/元素）→ 占用~40MB（节省50%）
arr32 = arr64.astype(np.float32)
print(f"float32内存: {sys.getsizeof(arr32) + arr32.nbytes:.2f}MB")  # ~38.1MB

# 图像处理场景：uint8（1字节/元素）替代int32（4字节/元素）
frame = cv2.imread("test.jpg")
frame_int32 = frame.astype(np.int32)  # 反例：内存×4
frame_uint8 = frame.astype(np.uint8)  # 正例：最小内存（默认）
```

### 2. 避免NumPy数组拷贝（复用内存）
工程落地：共享内存读帧默认零拷贝（视图），仅在注入缓存或跨生命周期持有时显式 copy，避免无意义的 `frame.copy()`。

#### 2.1 主链路实践：cv_worker 的“零拷贝”读取策略
在 Windows（spawn）多进程下，我们通过 `SharedMemory` 把帧数据从主进程传到 CV worker。**零拷贝**的关键是：worker 侧读取时返回指向 `shm.buf` 的 NumPy 视图，而不是立刻 `.copy()`。

- 统一入口：`cv_worker.get_frame_from_shm(shm_ref, copy=False)`
  - 做什么：`np.ndarray(shape, dtype=dtype, buffer=shm.buf)` 构造数组视图
  - 为什么：避免每帧一次内存分配与拷贝，显著降低峰值 RSS
  - 权衡：视图依赖共享内存生命周期，**不能**在共享内存可能被释放/复用的场景长期持有

工程规范（强约束）：
1) **短生命周期计算（推荐零拷贝）**：只在当前函数/当前批次内使用的帧，直接使用视图，不要 `copy()`。
2) **注入缓存/跨阶段持有（必须 copy）**：任何要放入 `_frame_cache`、LRU、或对象字段长期持有的帧，必须显式 `copy()`，避免共享内存被回收后出现悬空引用或数据被覆盖。
3) **及时释放**：任务结束后 `del frames/timestamps` 并 `gc.collect()`，缩短大数组驻留时间。

示例（概念代码）：
```python
# 1) 零拷贝：返回视图（frame 引用 shm.buf）
frame = get_frame_from_shm(shm_ref, copy=False)

# 2) 需要缓存时再拷贝：把数据从共享内存视图变成独立数组
validator._frame_cache[frame_idx] = frame.copy()
```
```python
# 反例：copy()会创建新数组，新增内存
arr_copy = arr32.copy()

# 正例：切片是视图（view），共享内存，无新增占用
arr_view = arr32[:5000000]  # 内存占用≈0
```

### 3. 及时释放无用数组
工程落地：CV worker 在任务结束后显式 `del frames/timestamps` 并 `gc.collect()`，降低峰值内存驻留时间。
```python
def process_numpy():
    big_arr = np.random.rand(10000000)  # ~80MB
    processed = big_arr.mean()
    
    # 显式删除+手动GC，立即回收内存
    del big_arr
    import gc
    gc.collect()
    
    return processed
```

## 三、Numba JIT（50-80MB）：减少编译缓存浪费
### 1. 只对核心函数用JIT，禁用不必要优化
```python
from numba import jit
import numba

# 仅对核心热点函数用JIT，且开启缓存避免重复编译
@jit(nopython=True, cache=True, fastmath=False)
def core_cv_func(x):
    """仅核心计算逻辑用JIT"""
    return x * 2

# 非核心函数不用JIT，节省编译缓存内存
def simple_func(x):
    return core_cv_func(x) + 1

# 限制Numba线程数，减少内存膨胀
numba.set_num_threads(1)
```

### 2. 清理Numba编译缓存
```python
import numba
# 清理长期运行产生的编译缓存（按需执行，比如每天1次）
numba.caching.clear_cache()
```
**效果**：可减少Numba内存占用10-20MB（约25%）。

## 四、CVKnowledgeValidator 缓存（60-100MB）：限制大小+淘汰策略
缓存命中率已统一接入统计与落盘，便于判断“哪些缓存值得保留”：
- 统一收敛器：`cache_metrics`，提供 hit/miss/snapshot/reset。
- 默认任务级复位，最终写入 `outputDir/intermediates/cache_metrics.json`（追加数组）。
- 输出结构：`{task_id, stage, ts, caches:{name:{hits,misses,hit_rate}}}`。
- 开关配置：`MODULE2_CACHE_METRICS_ENABLE=0` 关闭统计，`MODULE2_CACHE_METRICS_RESET_ON_TASK=1` 控制任务级复位。
### 1. LRU缓存限制最大条目
```python
from functools import lru_cache

# 限制缓存最多1000条，超出自动淘汰最久未使用的
@lru_cache(maxsize=1000)
def cv_validate(feature):
    """缓存核心验证结果，避免重复计算"""
    # 你的验证逻辑
    return True
```

### 2. 改用磁盘缓存（内存→磁盘）
```python
from diskcache import Cache
import time

# 缓存存储到磁盘，内存仅保留最近使用的条目
cache = Cache('./cv_cache', size_limit=100*1024*1024)  # 磁盘缓存上限100MB

def cv_validate(feature):
    key = hash(str(feature))  # 生成唯一key
    if key in cache:
        return cache[key]
    
    # 计算验证结果
    result = True
    # 存入缓存并记录时间（用于过期清理）
    cache.set(key, result, expire=3600)  # 1小时过期
    return result

# 定期清理过期缓存（每小时执行1次）
def clean_expired_cache():
    cache.expire()  # 自动删除过期条目
```
**效果**：内存缓存占用从60-100MB降至10-20MB（节省80%）。

## 五、帧处理缓冲区（50-100MB）：限制大小+压缩存储
### 1. 限制缓冲区帧数（FIFO策略）
```python
from collections import deque
import cv2

# 限制缓冲区最多10帧，超出自动弹出最旧的
frame_buffer = deque(maxlen=10)

def add_frame(frame):
    """新增帧到缓冲区，自动控制大小"""
    frame_buffer.append(frame)
```

### 2. 压缩帧数据（内存减少70-90%）
缓冲区存储压缩后的JPEG，而非原始像素数组：
```python
def compress_frame(frame):
    """压缩帧为JPEG，内存占用减少80%"""
    encode_param = [int(cv2.IMWRITE_JPEG_QUALITY), 80]  # 80%质量，平衡大小/画质
    _, encoded = cv2.imencode('.jpg', frame, encode_param)
    return encoded

def decompress_frame(encoded):
    """处理时解压，无性能瓶颈"""
    frame = cv2.imdecode(encoded, cv2.IMREAD_COLOR)
    return frame

# 缓冲区存储压缩后的帧
compressed_buffer = deque(maxlen=10)
compressed_buffer.append(compress_frame(frame))  # 单帧内存从~10MB→~2MB
```

### 3. 复用缓冲区内存
```python
# 预分配固定大小的缓冲区，复用内存（避免频繁创建新数组）
buffer = np.zeros((720, 1920, 3), dtype=np.uint8)  # 固定分辨率

def process_new_frame(new_frame):
    # 将新帧写入预分配缓冲区（无新内存分配）
    buffer[:new_frame.shape[0], :new_frame.shape[1], :] = new_frame
    return buffer
```

### 总结
针对你列出的模块，内存优化核心要点：
1. **OpenCV（最大瓶颈）**：多进程用`fork`/进程池复用，避免重复加载；Windows用共享内存，禁用OpenCL/可选禁用优化路径，单进程内存可降15%，多进程可省75%。
2. **数据层（NumPy/帧缓冲区）**：用紧凑数据类型（float32/uint8）、共享内存零拷贝、限制缓冲区大小、压缩帧数据，内存可降50-80%。
3. **缓存/编译层（CVKnowledgeValidator/Numba）**：设置LRU缓存上限、改用磁盘缓存、只对核心函数用JIT；并统一缓存命中率统计落盘，便于持续优化。
4. **通用原则**：限制进程数、及时释放无用对象、避免内存拷贝，整体可将单进程内存从400-580MB降至150-200MB（节省50%以上）。
