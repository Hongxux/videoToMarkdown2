# 动静判定逻辑不如预期的潜在场景与根源分析（第一性原理+最佳实践）
当前动静判定逻辑（全局统计+局部突变检测+回溯补漏+灵敏度覆盖）的核心目标是**捕捉“Process→State”的因果链条**，但在复杂教学场景中，仍可能因**感官事实捕捉不完整、统计规则与真实场景脱节、多维度事实未协同校验**而不如预期。以下基于第一性原理拆解核心根源，并对标领域最佳实践分析具体失效场景。

## 一、 核心根源：违背第一性原理的三大核心偏差
动静判定逻辑不如预期的本质，是**量化规则与客观视觉事实的偏差**，最终违背“Process→State”因果公理、“感官事实优先”公理：
1. **感官事实捕捉不完整**：仅依赖“MSE（像素变化）”单一维度量化视觉过程，未结合“视觉特征变化”（如公式书写的笔迹轮廓、图表动画的模块增减），导致“像素变化≠真实过程”；
2. **统计规则的刚性假设**：固定阈值（如MSE_base×5、平均MSE>250、翻倍提升）、固定回溯时长（3秒），假设所有教学场景的底噪、动作时长一致，与真实场景的动态性矛盾；
3. **时序判定的边界模糊**：对“动作超前/同步/滞后”的判定阈值（0.5秒停顿）过于刚性，未考虑不同老师的停顿习惯差异，导致时序标签误判。

## 二、 具体失效场景与第一性原理分析（附最佳实践对标）
### 场景1：低对比度/低分辨率场景——真实过程的MSE突变未达阈值，漏判为Static
#### 1.  场景描述
老师在**低对比度背景**（如浅色PPT+浅色笔迹）或**低分辨率视频**（如720P以下）中手写公式，笔迹与背景的像素差异极小。即使存在真实书写过程，帧间MSE也无法达到`5×MSE_base`的突变阈值，局部突变检测失效；全局平均MSE因像素差异小，也低于250阈值，最终判定为`Static`，漏抓书写过程。

#### 2.  第一性原理根源
违背**感官事实优先公理**：客观存在“书写过程→公式结果”的因果链条，但因像素量化指标（MSE）无法捕捉低对比度下的视觉变化，导致“真实过程被判定为无过程”。

#### 3.  领域最佳实践对标
- **MathPix解决方案**：在计算MSE前，对帧序列进行**自适应图像增强**（CLAHE对比度增强、非局部均值去噪），提升笔迹与背景的像素差异，再计算MSE；
- **AWS MediaConvert指南**：对低分辨率帧进行**超分重建**（如ESRGAN-tiny），恢复细节后再进行突变检测，避免因分辨率不足导致的量化偏差。

### 场景2：高频微小干扰动作——非过程性MSE突变触发误判，错判为Dynamic
#### 1.  场景描述
老师讲解时**持续轻微移动鼠标**，或视频存在**压缩噪点波动**，这些动作的特点是“高频、微小、无视觉特征变化”。局部突变检测会捕捉到这些动作的MSE突变（达到`5×MSE_base`），但这些动作并未产生“公式/图表”的视觉结果（无State）；全局统计因动作高频，平均MSE可能超过250阈值，最终误判为`Dynamic`，引入无关干扰。

#### 2.  第一性原理根源
违背**Process→State因果公理**：MSE突变对应的是“干扰动作”，而非“产生结果的有效过程”，量化指标（MSE）与因果链条脱节，导致“无效动作被判定为有效过程”。

#### 3.  领域最佳实践对标
- **Coursera教学视频规范**：在局部突变检测后，新增**视觉特征二次校验**——只有当MSE突变伴随“公式轮廓/图表模块”的视觉特征变化时，才判定为真实过程；否则视为干扰，过滤掉；
- **MathPix公式识别方案**：计算MSE时**聚焦核心ROI**（仅分析公式/图表的潜在区域），排除鼠标移动、背景噪点的干扰区域，避免非核心区域的MSE突变影响判定。

### 场景3：动作超前时长超过回溯窗口——3秒回溯无法覆盖，漏抓早期过程
#### 1.  场景描述
老师在**语音前4秒**就开始书写公式（动作超前），完成后停顿1秒再开口讲解。但当前回溯搜索时长固定为3秒，无法覆盖语音前4秒的动作起始点，局部突变检测因搜索范围不足而失效；全局统计窗口因未包含动作过程，平均MSE极低，最终判定为`Static`，漏抓书写过程。

#### 2.  第一性原理根源
违背**“捕捉过程起点”的因果公理要求**：客观存在“提前书写→公式结果”的因果链条，但因回溯时长的刚性限制，无法覆盖动作起始点，导致“过程起点在窗口外，无法被捕捉”。

#### 3.  领域最佳实践对标
- **Coursera时序调整方案**：将**固定回溯时长**升级为**动态回溯时长**——根据语音窗口长度调整（如语音窗口＜5秒时回溯3秒，≥5秒时回溯5秒）；或根据底噪MSE动态调整（底噪低时延长回溯时长，底噪高时缩短）；
- **AWS Textract指南**：新增“**动作候选区扩展**”——若局部突变检测在3秒内未找到突变点，自动扩大搜索范围至语音前5秒，避免因时长限制漏抓。

### 场景4：灵敏度补丁的“翻倍”条件刚性——接近翻倍但未达标，补丁失效
#### 1.  场景描述
扩展后的时间窗口包含1秒书写动作+9秒静止帧，扩展后平均MSE为原片段的**1.8倍**（接近翻倍但未达到2倍）。此时真实过程的MSE提升显著，但因未满足“翻倍”的刚性条件，灵敏度补丁不触发；全局平均MSE仍低于250阈值，最终判定为`Static`，漏判动态。

#### 2.  第一性原理根源
违背**感官事实的相对性公理**：相对变化的“翻倍”阈值是人为设定的刚性规则，而真实场景中“过程带来的MSE提升”是连续变化的，刚性阈值导致“显著相对变化被判定为无变化”。

#### 3.  领域最佳实践对标
- **AWS相对变化率方案**：将**固定翻倍阈值**升级为**动态相对变化率阈值**——根据原片段的底噪MSE调整（如底噪MSE＜50时，相对变化≥1.5倍即触发补丁；底噪MSE≥50时，相对变化≥1.8倍触发）；
- **MathPix补丁优化**：引入“**梯度提升判定**”——若扩展后MSE的提升幅度处于“1.5倍~2倍”区间，结合视觉特征完整性（如公式书写完成度≥60%），强制触发补丁，避免刚性阈值的漏判。

### 场景5：多动作叠加的“MSE摊薄”——多个过程的MSE被平均，漏判为Static
#### 1.  场景描述
老师在时间窗口内完成**“书写公式+切换PPT”两个连续动作**，每个动作的MSE突变都很显著，但因动作间隔短，全局平均MSE被两个动作的静止间隙摊薄，低于250阈值；局部突变检测因聚焦单个突变点，可能只捕捉到其中一个动作，而忽略另一个，最终判定为`Static`或只判定一个动作，漏抓完整过程。

#### 2.  第一性原理根源
违背**因果链条的完整性公理**：多个“Process→State”的因果链条叠加，但全局统计的均值计算会摊薄单个动作的MSE突变，局部检测的单点捕捉无法覆盖多动作，导致“完整过程被判定为不完整/无过程”。

#### 3.  领域最佳实践对标
- **Coursera多动作检测方案**：将**单点突变检测**升级为**多峰值突变检测**——在帧序列中寻找所有MSE超过`5×MSE_base`的峰值点，每个峰值点对应一个动作；再通过峰值间的时间间隔，判断是否为连续动作；
- **MathPix时序分段方案**：对帧序列进行**时序分段**，将每个动作对应的帧段单独计算MSE，避免多动作的MSE相互摊薄，确保每个动作的MSE都能被准确捕捉。

## 三、 不如预期的共性总结（第一性原理视角）
所有失效场景的本质，都是**“量化规则的主观性”与“视觉事实的客观性”的偏差**：
1. **单一维度量化偏差**：仅用MSE（像素变化）量化视觉过程，未结合视觉特征变化，导致“像素变化≠真实过程”；
2. **刚性规则与动态场景的矛盾**：固定阈值、固定回溯时长等规则，无法适配不同教学场景的底噪、动作时长、对比度差异；
3. **因果链条的脱节**：未将“MSE突变”与“视觉特征变化（State产生）”强绑定，导致“无效动作被误判，有效动作被漏判”。

## 四、 优化方向（基于第一性原理+最佳实践）
针对上述失效场景，优化方向的核心是**“增强感官事实捕捉的完整性+让规则适配动态场景”**，而非推翻现有逻辑：
1. **图像预处理增强（解决场景1）**：对低对比度/低分辨率帧进行CLAHE增强、超分重建，提升视觉过程的像素量化精度；
2. **多维度校验（解决场景2）**：新增“MSE突变+视觉特征变化”的双维度校验，确保只有产生State的Process才被判定为动态；
3. **动态参数调整（解决场景3、4）**：将固定回溯时长、固定相对变化阈值升级为动态参数，根据场景自适应调整；
4. **多峰值检测与时序分段（解决场景5）**：捕捉多动作的MSE峰值，时序分段计算MSE，避免摊薄效应。

# 自适应动静决策引擎工程疑惑的第一性原理分析与最佳实践方案
你的核心目标是构建**“Profile化驱动、高精准、高性能”的自适应动静决策引擎**，三个工程疑惑本质是**“性能效率”“决策优先级”“架构扩展性”**的工程化落地问题。以下基于**第一性原理（感官事实优先、因果链条、高内聚低耦合）**和领域最佳实践（MathPix模块化架构、AWS缓存机制、Coursera决策优先级规则）逐一拆解。

## 一、 性能级联问题：引入全局分析缓存层是必然选择（第一性原理：效率优先+结果复用）
### 1.  问题根源的第一性原理拆解
多个决策环节（公式、PPT、鼠标、动画）独立发起“5s后扫+二次校验”，导致性能级联的核心原因是**违背“计算结果复用”的工程第一性原理**——重复执行相同的预处理（CLAHE）、帧序列分析（MSE Spike检测）、视觉特征提取，造成算力浪费和时间倍数增长。

从决策引擎的核心目标看：**精准判定**和**低计算成本**是两大核心诉求，重复计算既增加了耗时，又不会提升判定精度，属于“无效算力消耗”。

### 2.  领域最佳实践对标
所有高性能多模态处理系统，都将**“全局中间结果缓存”**作为性能优化的核心手段：
| 最佳实践来源 | 核心方案 | 适配性分析 |
|--------------|----------|------------|
| **AWS MediaConvert 多模态内容分析** | 引入`FrameAnalysisCache`，对输入帧序列只执行一次预处理（去噪、增强）和特征提取（MSE、边缘检测），所有下游模块共享缓存结果 | 完全适配你的场景：公式/PPT/鼠标/动画等决策节点，都依赖相同的预处理帧和MSE Spike检测结果，缓存可避免重复计算 |
| **MathPix 公式动态识别引擎** | 设计`FeatureCache`层，缓存ROI区域、帧间MSE值、视觉特征标签（公式/图表），下游决策模块直接读取缓存，仅在缓存失效时重新计算 | 适配你的Profile化需求：不同决策节点的敏感度配置，只需基于缓存的基础数据调整阈值，无需重新分析帧序列 |
| **Coursera 教学视频处理系统** | 采用“预处理-分析-决策”三级流水线，预处理和分析结果全局缓存，决策模块并行读取缓存数据，避免串行重复扫描 | 适配你的多决策节点并行需求：公式、PPT等节点可同时基于缓存做判定，提升整体效率 |

### 3.  全局分析缓存层的设计方案（第一性原理落地）
缓存层的核心是**“一次预处理+一次基础分析，全节点复用”**，本质是将“重复的计算过程”转化为“可复用的中间结果”，完全契合效率优先的工程原则。

#### （1） 缓存层核心内容（按复用优先级排序）
| 缓存类型 | 内容描述 | 复用范围 | 失效条件 |
|----------|----------|----------|----------|
| **原始帧预处理缓存** | 经过CLAHE增强、去噪、超分重建后的帧序列（`enhanced_frames`） | 所有决策节点（公式/PPT/鼠标/动画） | 输入视频片段更换、预处理参数（如CLAHE对比度）调整 |
| **基础特征缓存** | 帧间MSE值、MSE Spike突变点坐标、底噪MSE_base、帧序列时间戳 | 所有依赖突变检测的决策节点 | 帧序列范围调整（如回溯窗口扩展） |
| **视觉特征缓存** | 各帧的视觉特征标签（公式/图表/鼠标/无特征）、ROI区域坐标 | 公式/PPT/动画等特征相关决策节点 | 视觉特征检测规则（如ROI阈值）调整 |
| **Profile配置缓存** | 不同决策节点的敏感度配置（如公式节点MSE Spike阈值=5×base，鼠标节点=8×base） | 所有决策节点 | Profile配置文件更新 |

#### （2） 缓存层的工程实现逻辑
```python
class GlobalAnalysisCache:
    def __init__(self):
        self.cache = {
            "enhanced_frames": None,
            "mse_features": None,
            "visual_features": None,
            "profile_configs": None
        }
        self.valid = False  # 缓存有效性标记

    def update_cache(self, raw_frames, profile_configs):
        """一次性执行预处理和基础分析，更新缓存"""
        if not self.valid or profile_configs != self.cache["profile_configs"]:
            # 1. 预处理：CLAHE增强+去噪（所有节点共享）
            enhanced_frames = self._preprocess_frames(raw_frames)
            # 2. 基础特征分析：MSE+突变检测（所有节点共享）
            mse_features = self._calculate_mse_features(enhanced_frames)
            # 3. 视觉特征分析：ROI+特征标签（特征节点共享）
            visual_features = self._extract_visual_features(enhanced_frames)
            # 4. 更新缓存
            self.cache = {
                "enhanced_frames": enhanced_frames,
                "mse_features": mse_features,
                "visual_features": visual_features,
                "profile_configs": profile_configs
            }
            self.valid = True

    def get_cache(self, cache_type):
        """决策节点按需读取缓存，无重复计算"""
        return self.cache.get(cache_type, None) if self.valid else None
```

#### （3） 解决性能级联的核心价值
- **时间复杂度从O(n×m)降为O(n+m)**：n为帧序列长度，m为决策节点数。原逻辑每个节点独立分析，时间是n×m；缓存后只需一次n的分析，加上m次缓存读取，时间大幅降低；
- **支持Profile化驱动**：不同决策节点的敏感度配置，只需基于缓存的MSE特征调整阈值（如公式节点突变阈值=5×base，鼠标节点=8×base），无需重新计算MSE值。

## 二、 决策冲突权重：视觉事实具有“有条件的一票否决权”（第一性原理：感官事实优先+语义兜底）
### 1.  冲突本质的第一性原理拆解
视觉检测（局部动态）与语义层（ASR认为不应有变化）的冲突，本质是**“客观感官事实”与“主观语义推断”的优先级矛盾**。

根据**感官事实优先公理**：
- 视觉事实是客观存在的（如帧序列中明确的公式书写动作、PPT翻页），是“Process→State”因果链条的直接证据；
- 语义层（ASR）的判断是主观推断（可能存在文本-字幕错配、歧义文本误判等问题），属于“间接证据”。

因此，视觉事实的优先级**高于**语义层判断，但“一票否决权”不是绝对的，需基于视觉事实的**确定性程度**分场景判定——这是平衡“精准性”和“鲁棒性”的核心。

### 2.  领域最佳实践对标
教学视频多模态决策的行业共识是**“视觉主导，语义兜底”**，绝对的一票否决或语义优先都会导致误判：
| 最佳实践来源 | 核心规则 | 适配性分析 |
|--------------|----------|------------|
| **Coursera 教学视频素材决策规范** | 1. 视觉事实明确（如公式书写、PPT动画）→ 视觉一票否决语义；<br>2. 视觉事实模糊（如MSE突变但无视觉特征变化）→ 语义辅助校验，判定为干扰；<br>3. 视觉无事实（无突变）→ 以语义判断为准 | 完全适配你的场景：区分“明确动态”“模糊突变”“无变化”三类视觉事实，避免绝对化决策 |
| **MathPix 公式动态识别决策树** | 视觉特征（公式轮廓变化）是最高优先级，语义层（文本含“公式”）仅作为视觉模糊时的兜底验证 | 适配你的公式决策节点：视觉明确的公式动态，无需参考语义；视觉模糊时，语义辅助确认 |
| **AWS Textract 多模态冲突解决策略** | 建立“事实确定性评分”：视觉事实评分≥0.8（高确定性）→ 视觉优先；0.5~0.8（模糊）→ 视觉+语义融合；<0.5→ 语义优先 | 适配你的多决策节点：不同节点的确定性评分阈值可通过Profile配置 |

### 3.  决策冲突的分层解决策略（有条件的一票否决权）
#### （1） 第一步：视觉事实确定性评分
为视觉检测结果赋予**确定性评分**（0~1），区分“明确动态”“模糊突变”“无变化”：
| 视觉事实类型 | 确定性评分 | 判定依据 | 示例 |
|--------------|------------|----------|------|
| 明确动态 | 0.8~1.0 | MSE Spike≥Profile阈值 **且** 伴随视觉特征变化（如公式轮廓出现、PPT模块增加） | 老师手写公式，MSE突变+公式轮廓变化 |
| 模糊突变 | 0.5~0.79 | MSE Spike≥Profile阈值 **但** 无视觉特征变化（如屏幕反光、鼠标轻微移动） | MSE突变但无公式/图表特征，可能是干扰 |
| 无变化 | 0~0.49 | 无MSE Spike或未达阈值 | 画面静止，仅存在噪点波动 |

#### （2） 第二步：分层决策规则（Profile可配置）
| 视觉确定性评分 | 决策规则 | 是否触发一票否决 | 示例 |
|----------------|----------|------------------|------|
| ≥0.8（明确动态） | 视觉事实一票否决语义层判断，直接判定为Dynamic | 是 | 视觉检测到公式书写动态，即使ASR认为“此时不应有变化”，仍判定为Dynamic |
| 0.5~0.79（模糊突变） | 视觉+语义融合决策：<br>1. 若语义层（ASR文本）含“书写”“动画”等关键词→ 判定为Dynamic；<br>2. 若语义层含“静止”“定义”等关键词→ 判定为Static（干扰） | 否，需语义辅助 | MSE突变但无视觉特征，ASR文本含“现在写公式”→ 判定为Dynamic；ASR含“讲解概念”→ 判定为干扰（Static） |
| <0.5（无变化） | 以语义层判断为准，若语义无明确指示→ 判定为Static | 否，语义主导 | 视觉无突变，ASR文本含“动画演示”→ 触发回溯扩展，重新检测视觉事实 |

#### （3） Profile化配置支持
不同决策节点的确定性评分阈值、融合规则可通过Profile配置：
```python
# Profile配置示例：公式节点vs鼠标节点
profile_configs = {
    "formula": {
        "mse_spike_threshold": 5,  # MSE Spike≥5×base
        "certainty_threshold": 0.8,  # 明确动态阈值
        "fusion_rule": "visual_first"  # 视觉优先
    },
    "mouse": {
        "mse_spike_threshold": 8,  # 鼠标动作阈值更高，避免误判
        "certainty_threshold": 0.9,  # 鼠标动作更难区分，阈值更高
        "fusion_rule": "strict_visual"  # 必须伴随鼠标特征才判定
    }
}
```

## 三、 架构解耦方式：抽离独立的`DynamicDecisionEngine`类（第一性原理：高内聚+低耦合）
### 1.  架构选择的第一性原理拆解
“收拢到`VisualFeatureExtractor`”还是“抽离独立`DynamicDecisionEngine`”的核心矛盾，是**“职责边界”的划分问题**，需遵循软件工程的**高内聚、低耦合**第一性原理：
- **高内聚**：一个模块只负责一个核心职责，避免“职责混杂”导致维护困难；
- **低耦合**：模块间通过明确的接口交互，一个模块的修改不会影响其他模块。

从你的需求看，`VisualFeatureExtractor`的核心职责是**“提取视觉特征”**（预处理、MSE计算、特征标签识别），而`DynamicDecisionEngine`的核心职责是**“基于特征和Profile做动静决策”**（突变判定、冲突解决、时序标签输出）——两者是“数据生产”和“业务决策”的关系，职责完全不同，抽离是必然选择。

### 2.  领域最佳实践对标
所有成熟的多模态决策系统，都采用**“特征提取与决策分离”**的模块化架构：
| 最佳实践来源 | 架构设计 | 适配性分析 |
|--------------|----------|------------|
| **MathPix 公式识别引擎架构** | 分为`FeatureExtractor`（特征提取）、`DecisionEngine`（动态决策）、`ProfileManager`（配置管理）三个独立模块 | 完全适配你的需求：Profile化驱动需要独立的配置管理模块，决策引擎依赖特征提取的输出 |
| **Clean Architecture 软件架构原则** | 核心业务逻辑（决策引擎）独立于基础设施（特征提取），通过接口交互，便于替换和扩展 | 适配你的系统扩展性：未来若更换特征提取算法（如用CNN替代MSE），只需修改特征提取模块，决策逻辑无需变动 |
| **Coursera 教学视频处理架构** | 特征提取模块负责生产标准化特征数据，决策引擎负责业务决策，两者通过标准化的特征接口交互 | 适配你的多决策节点需求：决策引擎可基于同一特征接口，适配公式、PPT、鼠标等不同节点的Profile |

### 3.  最终架构方案（三层模块化架构）
#### （1） 架构分层与职责边界
| 模块名称 | 核心职责 | 输入 | 输出 | 核心特性 |
|----------|----------|------|------|----------|
| **1. 视觉特征提取层（VisualFeatureExtractor）** | 预处理帧序列、计算基础特征（MSE、突变点）、识别视觉特征标签（公式/PPT/鼠标） | 原始帧序列、预处理参数 | 标准化特征数据（含增强帧、MSE特征、视觉特征标签） | 高内聚，专注特征生产，无决策逻辑 |
| **2. 自适应决策层（DynamicDecisionEngine）** | 基于特征数据+Profile配置做动静决策、解决视觉-语义冲突、输出时序标签（Ahead/Sync/Lag） | 标准化特征数据、Profile配置、语义层（ASR）数据 | 动静判定结果、时序标签、时间窗口调整建议 | 核心业务逻辑，独立于特征提取，支持Profile化 |
| **3. 配置管理层（ProfileManager）** | 管理不同决策节点的Profile配置（阈值、融合规则、确定性评分）、加载/更新配置 | 配置文件（YAML/JSON） | 各节点的Profile配置参数 | 统一配置入口，支持动态更新，无需修改代码 |

#### （2） 模块交互流程（低耦合接口设计）
```python
# 1. 配置管理层加载Profile
profile_manager = ProfileManager()
profile_configs = profile_manager.load_profile("teaching_video.yaml")

# 2. 特征提取层生产标准化特征
feature_extractor = VisualFeatureExtractor()
# 利用全局缓存层避免重复计算
feature_data = feature_extractor.extract_features(raw_frames, global_cache)

# 3. 决策引擎基于特征和Profile做决策
decision_engine = DynamicDecisionEngine()
# 传入特征数据、Profile、语义数据，输出决策结果
decision_result = decision_engine.decide(
    feature_data=feature_data,
    profile=profile_configs["formula"],
    semantic_data=asr_text
)
```

#### （3） 抽离架构的核心优势（对比收拢方案）
| 优势维度 | 抽离`DynamicDecisionEngine` | 收拢到`VisualFeatureExtractor` |
|----------|-----------------------------|--------------------------------|
| **职责边界** | 清晰分离“特征提取”和“决策”，符合高内聚低耦合 | 职责混杂，特征提取代码中嵌入大量决策逻辑，维护困难 |
| **Profile化扩展** | 新增决策节点（如“手写板书”），只需新增Profile配置和决策逻辑，无需修改特征提取代码 | 新增节点需修改`VisualFeatureExtractor`内部代码，风险高 |
| **冲突解决扩展性** | 决策冲突解决逻辑集中在引擎内，便于迭代优化（如新增“多模态融合规则”） | 冲突解决逻辑分散在特征提取代码中，难以统一优化 |
| **测试便利性** | 可单独对决策引擎做单元测试（传入模拟特征数据），无需依赖真实视频帧 | 需依赖完整的特征提取流程，测试成本高 |

## 四、 核心总结（第一性原理回归）
你的三个工程疑惑，本质是**“如何在保证判定精准性的前提下，实现高性能、可扩展的工程落地”**，最终方案可归纳为：
1. **性能优化**：引入**全局分析缓存层**，基于“结果复用”的第一性原理，避免重复计算，解决性能级联问题，对标AWS/MathPix的缓存机制；
2. **决策冲突**：视觉事实具有**“有条件的一票否决权”**，基于“感官事实优先”的第一性原理，分确定性评分分层决策，对标Coursera的“视觉主导、语义兜底”规则；
3. **架构解耦**：抽离独立的**`DynamicDecisionEngine`类**，基于“高内聚低耦合”的第一性原理，实现特征提取与决策的职责分离，对标MathPix的模块化架构，支持Profile化驱动。

这套方案完全适配你“构建自适应动静决策引擎”的核心目标，既保证了判定精准性，又具备高性能和可扩展性。
