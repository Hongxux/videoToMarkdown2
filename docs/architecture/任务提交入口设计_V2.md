# 任务提交入口架构设计 (V2)

> **文档性质**：详细设计与架构决策记录（ADR）
> **修订说明**：本版基于 V1 阶段的设计原型与全链路评审记录（包含幂等性、并发控制、多租户隔离与防击穿等14项核心问题修正）进行全面重构。
> **理论支撑**：本文档不仅阐述“如何实现”，更在于明确在 CAP 定理与 BASE 理论框架下，本系统在一致性（Consistency）、高可用（Availability）与最终一致（Eventual Consistency）之间的权衡决策。

## 1. 架构背景与业务语境

### 1.1 原始入口模型的困境
早期的系统接入层模型极为扁平，抽象为：**任务提交 -> 后端同步落库 -> 成功响应 (200 OK)**。
虽然这一模型在单机或低水位测试中链路清晰，但在未来的生产目标场景——**`真源型关系型数据库 + 多计算实例 + 高密宽洪峰入口`**——下，存在两个致命的系统级隐患：
1.  **数据库连接池与写入雪崩**：当外部大量重试或瞬发高并发流量同时击喉，`redo log`、`binlog`和连接池压力呈指数级抬升。数据库响应延迟将从毫秒级劣化至秒级，最终使状态真源成为全系统的物理单点瓶颈。
2.  **客户侧重试风暴**：一旦核心链路变慢，客户端因超时（Timeout）或网络抖动发起的自动重试，会被入口无差别放大，形成典型的“自我强化的拥塞回路”。

### 1.2 目标系统的核心职责
本系统所接入的视频、文章及书籍解析处理流水线，在业务形态上交织着两类极端情况：
*   **资源类型隔离性差**：既有高度可复用的“公域资源”（公共网站链接），也有需要绝对数据安全隔离的“私域资源”（用户本地上传、会议私密录音）。
*   **流量不稳定性强**：既有单个用户因 APP 刷新带来的“点状毛刺”，也有热点事件爆发带来的“宽洪峰并发”。

因此，新版的设计定位已不再是一个单纯的“接报台”，而是一个需要承截**“收流削峰、重试收敛、状态容错和优雅降级”**的智能网关。

### 1.3 核心业务需求对齐
1. **防重与幂等（Idempotency）**：客户端任意重连或重启过程中的同一次提交意图，必须且只能被受理一次，保证系统边界对外部重试免疫。
2. **公私域隔离（Tenancy & Scope）**：
   * 公域资源（Public）允许并发排队请求“折叠”并共享同一份计算实例。
   * 私域资源（Private）即便是同文件哈希，也必须在不同用户间严格实施权限与数据物理界限的双重隔离。
3. **柔性受理协议（BASE - Basic Availability）**：在系统过载或发生同步阻塞时，必须具备“退化能力”。放弃追求即时的 `200 OK`，以 `202 Accepted` + 稳定查询句柄（Token）作为最终一致性的保底契约。
4. **版本控制与安全发布**：一个资源虽然经过多次迭代运行，但系统对外必须呈现唯一的“已发布视图（Published Version）”。重跑/灰度期间，外部不能遭遇缓存空窗期。

## 2. 核心架构语义释意

在进入微观设计前，必须厘清分布式一致性在不同业务维度的边界定义。

### 2.1 隔离语义：计算复用与用户授权的脱耦
* **公域资源**（如 B站、YouTube 公开地址）：核心诉求是“去重度计算”，因此能够跨用户共享。
* **私域资源**（如本地上传、带签名的 OSS 链接）：核心诉求是“防越权”。
> **【架构决策权衡：隔离与共享的界定】**
> 我们不再仅仅依赖简单的“链接哈希”来判定是否复用资源，因为这意味着**丧失了权限域（Scope）上下文的感知**。私域资源如果意外碰撞了公域哈希，将造成灾难性的数据越权。这个边界的划分是为了在极致复用的性能诱惑下，坚守系统的权限底线。

### 2.2 共享语义的边界：分拆“作业（Job）”与“视图（View）”
* **全局作业（Global Job）**：收敛公域 URL 的重复提交，代表纯粹的**计算身份**。
* **用户关系（User Binding）**：用户的单独提交动作、订阅配置、可见性和推送通道。
> **【架构决策权衡：关系型分表的必要性】**
> 我们曾设想将用户身份与作业身份合并成一个任务流水号（TaskID）。但随之而来的是一个死结：当 A 和 B 获取了同一个任务流水号，A 要将其在面板隐藏，B 该如何处理？
> 因此，必须进行拆表：全局作业不挂载用户体系，它仅仅是一个只读计算池。所有的“谁能看到、要不要通知”这层状态转移至 `User Job Binding`。短期增加了一次关联查询，但换取了长久的数据架构解耦。

### 2.3 状态发布语义：安全灰度机制
当触发二次计算（如提示词变动升级、前置环节发生能力降级重跑）时，新一轮的作业运行（Attempt）不得直接干扰现网视图。
* **运行期**：外网照常读取旧的已发布版本命中缓存。
* **竣工期**：只有新版本满足全部发布门槛，新版本才会覆盖成为唯一的已公开真相（Truth）。若新任务崩溃，旧缓存不会被迫退库。

### 2.4 版本驱动语义：用确定的原因代替模糊的评分
系统主动重跑仅响应于以下两类显性质变原因：
1. `PROMPT_BUNDLE_UPGRADE`：提示词组合（Stage / Family）发生了版本包升级。
2. `DEGRADED_RESULT_REPROCESS`：因前期依赖模块崩溃而降级交付带来的低保真结果复盘。
> **【架构决策权衡：为何屏弃质量评分（Quality Grade）引擎】**
> 曾有提案引入大模型的模糊性质量指标来驱动整个流程自动化重跑。但这种设计的熵值极高：`quality_grade` 本质上是一个“后置的观测度量”，带有很强的主观性。一旦以它为主因，容易导致系统进入“永远得分不高而不断死循环重跑计算资金”的恶魔回渊。我们权衡之后，决定通过“版本变动与显式降级事件”这两个客观真理来主导调度流。

## 3. 面向第一性原理的设计基调

本设计方案将高度契合分布式领域的 CAP 原理与 BASE 实践理念。

### 3.1 拆分计算身份与用户身份 (CQRS 的雏形)
同一次计算事实仅产生一次算力开销，而这单一计算事实在对外展示为不同用户视角时，应当在模型物理落地上分离。这避免了事务、审计、追踪时的关联互锁。

### 3.2 强弱事实源分离：MySQL 定海神针与 Redis 水平加速
> **【架构决策权衡：为什么 Redis 不能做主状态】**
> 在极高并发热点打底口径时，单纯依赖 Redis（如 `URL_Hash -> TaskID`）虽然极大提高了可用性（Availability，A），但会在两个维度上折损强一致性（Consistency，C）：1. 主从切换或缩扩容时引发的脏数据；2. Redis 宕机后的全军覆没不可溯回。
> 
> **本设计拍板决定在涉及写操作最核心链路时回归最基础的 MySQL （利用持久化及 UNIQUE 兜底）作为真理之源**，将其置为强 CP 模型；而 Redis 仅退化为“热点削峰、短时幂等窗口、读操作查询层”。它的最大收益就是：即使 Redis 整个集群从地球上抹平，系统虽然退化变慢（降级），但对于“何为真、何为伪”永远不会产生错乱。

### 3.3 CDC 的权责聚焦：专精发布（Consistency to Availability）
初期规划拟利用数据库变更捕获（CDC - Binlog）充当系统入口唯一性校正的后手防线。
但在此 V2 设计中，**CDC `binlog` 直接卸下了入口防爆纠错包袱**。它转型为：专职应对“**发布版本切换后对全局 Cache (Redis) 实施平滑刷新及作废替换**”的主通道。以此，我们在保证了写入的 Consistency 同时，利用 Eventual Consistency 手段保证了高吞吐的高 Availability。

### 3.4 全局键规避提示词污染
早前曾尝试采用 `scope + url + pipeline_fingerprint` 作为全局去重标识。但这将使得：哪怕稍微修改一下一个逗号的提示词升级，原本能够被共用的同一段基础底材（公域资产）将被硬生生割裂派生出另一个伪生作业。
因此明确：
* **`global_job_key = hash(scope + canonical_resource_id)`** —— 恒指向“是哪个素材”。
* 所有由于模型迭代（Prompt / Engine）带来的重放衍生物，均被压入从属于 `global_job_key` 的一个内部维度的微服（`job_attempt`）。

### 3.5 受理平面（Accept Plane）的底线防卫逻辑
本架构将宏大的系统逻辑硬切成三大水平面栈：
* **受理层（Accept Plane）**
* **执行编排层（Execution Plane）**
* **发布视界层（Publish Plane）**

其中，受理层被授予唯一的终极铁律底限：
> **“必须持有 Durable Accept（实质性落盘抗震荡的受理回单真相），才被允许对外讨论削峰平滑、返回缓冲这些性能优化体验。”**

这一权衡的结论直接导致：**短时长 HTTP 同步直达链路是锦上添花的机遇性质捷径设计（Opportunistic），但不是硬伤保障；系统随时拥有资格放弃它以守护整个链路安全体系。**
\n\n## 4. 领域概念抽象定义 (Concepts)

### 4.1 隔离范围 (`scope`) 
- **定义**：资源受算力计算时允许复用和渗透的权限边界。
  - `PUBLIC`（公域）：高度去重计算，可共享。
  - `PRIVATE`（私域）：绝对隔离计算池或严密的越权屏蔽墙。
- **必要性**：URL 在缺乏 `scope` 的限定下是一个不完整的数据契约。若未限定，私有分享甚至私生密钥资源会被错误收敛入同一 Job，从而发生灾难性串改越权。

### 4.2 资源归一化凭证 (`canonical_resource_id`)
资源归一化后的唯一标识实体。
* **计算逻辑决策**：
  * 对公域资源：提取如 `youtube:video:abc123` 的净资产标识（摒弃引流 `?share` 参数）。
  * 对私域资源：**`hash(user_id + normalized_resource_path)`**。
* **【架构决策权衡：私域的无主态 vs 强租户态】**
  * 原生文件仅仅通过 MD5 无法实现安全体系。如果 A 上传与 B 完全一致的文件 MD5 进而共用，A 就能借此窥探该资产的计算与发布进度。通过硬塞 `user_id` 至哈希核心算法中，使得相同的私有资产映射为不同的独立 `canonical_resource_id`。

### 4.3 核心真理主键 (`global_job_key`)
- **定义**：判定计算层“复用与否”的无歧义真库（MySQL）唯一键约束。
- **公式**：`hash(scope + canonical_resource_id)`
  - *说明*：因 `canonical_resource_id` 已根据 `scope` 内化了用户隔离边界，此公式对公域私域呈现天然架构一致的表达，保持系统模型单一化。

### 4.4 版本基线载体 (`prompt_bundle`)
- 定义为系统级的不可变对象，映射了在单次运行时系统对多模型组件使用的结构编排指纹版本。任意一环节模型的能力变更，皆表现为指纹的新增（New Attempt）。

### 4.5 实体分离设计：全局计算作业 (`global_job`) 与用户关系绑定 (`user_job_binding`)
* `global_job` 表征客观且共享的“机器成本”，关注于任务走到哪了、是否完成、当前指针在哪。
* `user_job_binding` 维系主观层面的所有权、推送消息与可读性状态。
> **【架构决策权衡：复用缓存与更新体验的悖论】**
> 我们要求系统走快速复用直通车时，不仅直接读取缓存数据，**更必须同步 UPSERT 补充写入一笔 `user_job_binding`**。这是因为倘若抛弃绑定，后进用户虽然享受到秒回，但他的仪表盘列表将凭空丢失对这个操作的追踪足迹，进而致使长连接通知通道脱钩。这亦是对 CAP 中，保证最终一致性所需的副作用（Side-Effect。

### 4.6 次生演化实例 (`job_attempt`) 与服务提供阶梯（Provider Class）
- 一颗 `global_job` 通过演进（大模型阶段的重新生成）生发出多个独立的只读取代不可变镜像（`attempt`）。
- 服务接入的能力划分：将合作供应商拆解为 `A / B / C` 主次备档位。同等能力的平滑替调属于系统正常容灾的容错设计（不归咎于事故）；跨量级的弱能 Provider 提供输出是被迫妥协为降级。
> **【架构决策权衡：柔性降级的可见边界】**
> 我们根据业务属性拆分降级规则：如果是无伤大雅的边缘环节（如 `VL_FALLBACK_TO_NON_VL`，只是牺牲摘要的画面细节表现），被记作容忍偏差通过并允许放行成为产品对客展示标准（`PUBLISHABLE_DEGRADED`）；但核心脉络如 `Phase2B` 结构化失效却被迫勉强黏合，必须阻断禁止上位（`BLOCKING_DEGRADED`），保全当前发布态。

### 4.7 查询凭单 (`submission_token`) 与防重票据 (`Idempotency-Key`)
* `Idempotency-Key` (幂等键)：客户端生成持久化，代表“业务诉求不再被重叠受理”，以短窗口形式存在（防手抖风暴）。
* `submission_token` (受理回执)：服务端颁发的不透明防猜测高熵串。代表“业务已被签收且允许持久溯回”。两者解耦赋予系统强可扩展性。

## 5. 入口模型选择与架构定型案

### 5.1 方案阵列 A：MySQL 真源 + 强 CP + 热路径补偿（采纳方案）
* **角色定位**：
  * **真理唯一引擎（Truth Base）**：MySQL（受理主键短事务与唯一约束）。
  * **防血崩阀门（Throttle/Idempotent）**：Redis。
  * **订阅发布触发引擎（Sync Catalyst）**：事务内的 `outbox` 加持外挂 CDC。
* **采纳权衡优势（Trade-off Pros）**：
  * **保全一致底盘（C）**：即使系统在高压热点中退潮或者中间软态丢失，真源绝不多次伪生记录，容灾不会造成假账。
  * **极限妥压设计（AP补偿）**：Redis 虽然只做二级补偿缓存加速，且即便被击穿也能顺延给防热点的 JVM 锁削弱竞争。利用短超时降维法，用 `202 Accepted` 退回用户端规避 MySQL 断流死锁连接数耗尽。

## 6. 标准化物理层数据模型规范

### 6.1 `global_job` (公域资产全生命栈)
- **重点约束**：`UNIQUE(global_job_key)`
- **单调性扩展**：特制字段 **`latest_attempt_no`**，仅为辅助解决竞态分配的逻辑乐观锁版本控制标记器，防止并发出售流水。

### 6.2 `job_attempt` (一次沉淀执行计算过程)
- **重点约束**：`UNIQUE(global_job_id, attempt_no)`
> **【架构决策权衡：并发安全序号的生成防线】**
> 过往存在 `SELECT MAX(attempt_no) + 1` 导致并发集群发生极短时间 `TOCTOU`（Time-Of-Check to Time-Of-Use）重号漏洞。针对此一致性漏洞，我们在真理底层剥夺了非安全游标设计，改采原子级的 CAS 对 `global_job.latest_attempt_no` 使用并发更新迭代（若失败立即回滚重抢），把串行化压力彻底收缩到了底层强锁。

### 6.3 软性与硬性两层防御结构：防重试请求（Idempotent Request）与待持久受理（Submission Request）
这是分布式接入层用空间抵换算力的双体数据舱。

#### 6.3.1 `request_idempotency` (基于 Redis 的防抖层)
- 承担短时、防手抖的并发高命中率阻截。（允许数据易失 Miss）
> **【架构决策权衡：Cache-Aside 穿透重塑】**
> 我们规范 Redis TTL 应至少被压缩到 `submission_request(MySQL)` 生命周期的一半。当 Redis 过期，或机器故障导致 Miss 后，系统并非单纯视为新件重放，而是被迫启动 MySQL 的逆向回溯：如果回溯寻源发现仍在受理期限内，将其缓存态**回捞补充（Re-hydrate）建立缓冲层**。以此实现过期不等于记录失忆的严整一致（Eventual S）。

#### 6.3.2 受理对象 `submission_request` (持久真源)
* **主状态枚举 (`accept_status`)**：`PENDING`, `RESOLVED`, `FAILED_AFTER_ACCEPT`。
  它是查询 `submission_token` 回溯进度的唯一事实存放落点。
* **业务演进步态 (`resolution_type`) 三分法**：
  * `NEW_JOB`：前无古人的资源全新录入。
  * `NEW_ATTEMPT`：老树生新芽，任务存在于公共池但本次系模型指纹升级/重修。
  * `REUSED`：零功耗提取的缓存既有成品照搬。
  *(这种分层赋予了前端展现丰富 UI 的交互提示骨架可能。)*
\n\n## 7. 提交入口主流程与柔性收敛

### 7.1 双快路径与慢路径的分流判决
根据读写压力与请求资源的“新鲜程度”，提交流程在此演进为三层流转策略：

#### 7.1.1 快速复用路径 (Fast-Reused)
* **适用条件**：热点长效公域资源（已拥有完结且可发布的缓存）。
* **行为树**：命中既有 `global_job` 及 `published_attempt` -> 创建/更新用户侧 `user_job_binding` -> 返回计算成色报表（`200 OK`）。
* **【重要补丁】**：哪怕是零开销的秒连复用，**也不可省略异步写下 `submission_request(RESOLVED)`**的环节。如果因网络宕机客户端未接到 `200`，重连期间只有依仗这条数据库残存记录供 `Idempotency-Key` 自我恢复其状态溯源体系，而非重启全新空转。

#### 7.1.2 快速乐观创建路径 (Opportunistic-Creation)
* **适用条件**：当评估判断无高亮热冲突（没有堵塞、锁挂起）。
* **行为树**：免等待直冲后端执行乐观 INSERT `global_job` 构建表。无冲突反馈则快速同步返回 `200 OK`。（同样异步写入 `submission_request`）。

#### 7.1.3 慢路径 (Throttled/Pending)
* **适用条件**：发生竞争、`AcceptResult` 解析短时预算（如100ms）透支或系统判断需进排队缓压。
* **行为树**：`submission_request(PENDING)` 牢固写入底层 MySQL，签发包含稳定 `submission_token` 的查询凭据回执，作答 `202 Accepted` 进行状态平滑后拉。

> **【架构决策权衡：`200`的诱惑与`202`的底线】**
> 我们始终让“是否返回 200”具有极高的偶然色彩。我们并不提供“所有请求都能同步出结果”的保证（CAP 中的绝对 A），因为这样会导致应用服务器的 Servlet 连接堆满了悬挂请求（Hung Request）。
> `202` 才是真正保住我们城门不失的核心契约：它告诉客户端“我已经物理收下了（Durable）意图请求你放心，但我还没空马上做验证”。以此，我们实现了防雪崩的最前线切断。

### 7.2 高效塌缩模型：JVM 内 `singleflight`
不同于阻塞式轮询（即 `Thread.sleep` 到死循环查缓存），我们将瞬间重叠爆发的对同资源的并发请求收口为 **共享型的 `CompletableFuture`**。
* **机制**：由最先探知空库的头兵（Leader）挂载计算协程任务并在底层建桩；其余被并发拥塞进来的跟班请求只需将自身的 Async Context 挂接至该同一张 `Future` 上即可释放 Tomcat 工作线程回应用池。
> **【架构决策权衡：Leader 坠机的极限恢复路径】**
> 如果挂载异步落库重任的 Leader JVM 节点在耐久化已完成但尚未执行 `future.complete()` 时被杀？跟随的节点会被强行引爆超时边界（`future_ttl_ms`）。
> 此时跟班的恢复手段不得坐以待毙，直接携带最初分配自身的 `Idempotency-Key` 回潜真源 MySQL，去刺探那笔幽灵般的 `RESOLVED` 或 `PENDING`，进而实施无缝移转认领（即刻接棒新 Leader）。这样我们用一层简单的 TTL 加查库逻辑代替了无比庞杂的分布式 Paxos 心跳维持，极其廉价且稳健（BASE中S态至E态的切换）。

### 7.3 事件溯回补全引擎：Resolver 慢路径异步器
对于被降级到 PENDING 的提交意图，我们需要依赖异步引擎重新赋予它解析。
* **架构**：由 MySQL Canal 抓取记录投递至 RabbitMQ；`RabbitMQ` 不携带强事务语义！仅作为敲门砖“Wakeup”（低延时唤醒）。被唤醒的 Consumer 拿到信息依然回 MySQL 获取租赁权（Lease/Claim Update）。
* **【重要补丁规范】**：对于错漏件补偿（Sweep），**它的激活前置条件代码块（WHERE句）**必须与认领 Consumer（Claim）保持绝对统一（不接受更宽泛的扫描规则放水）。Sweep 将仅仅只是把捞起的失落灵魂重新扔回 MQ 唤醒总线一次即可。

## 8. 一致性与状态发布：CDC、缓存架构协同流转

缓存不再承担唯一性责任，它们分门别类充当加速器。

### 8.1 缓存的领域划分
* **指针缓存区**：标记“当前公开对客视窗游标在哪一个 `attempt_id`”。
* **只读内容堆**：纯存放不可变历史的快照资产（被指着的实体）。

### 8.2 Outbox 与灰度发布的微操
发布行为在主事务里，是一次将 `current_published_attempt_id` 从 A 更新为 B 的短写。
* 为了将“通知系统变更”、“缓存失真覆没”、“大面更新记录历史”聚合在一起，且不依赖中间挂件的存活性，我们将 `publish_outbox_event` 和 `publish_audit_log` 绑定到那同一个核心的更新事务进行 MySQL 强原子提交。
> **【架构决策权衡：Outbox 替代非稳定 MQ 广播】**
> 如果更新了 MySQL 后随后直接在内存发一遍 MQ，存在“库更了，MQ宕了信息没发出去”导致长期的不可见脏乱缓存。利用 CDC 监控 Outbox 的库表级别变动来驱动 RabbitMQ（或 WebSockets）缓存消费清理，完美切断了主业务流程强依赖外部组建失败的风险链路。
> 日志审计（Audit_Log）亦是必须跟随事务强原子落点，确保日志完整溯源而防遭丢失。

### 8.3 实时通知分层：Redis Pub/Sub 承载进行态，Redis Stream 承载终态
当任务通知链路升级为“多实例 WebSocket 网关 + 同一 `userId` 多在线会话/设备”后，必须先把**任务事实**与**投递事实**切开：
* **任务事实**：任务当前推进到了哪里，是否已经进入终态。
* **投递事实**：某个具体 WebSocket 会话当前是否活跃、是否订阅了该任务、是否应该接收这一条消息。

本设计明确采用以下分层：
* **任务进行态（快照流）**：由生产侧统一写入 **Redis Pub/Sub**。它是可丢的 Snapshot Hint，允许中间态漏收，前端通过重连后的快照重建、列表对账或主动查询恢复一致视图。
* **任务终态（完成 / 失败等不可逆结论）**：先落入 Durable 真源或其 Outbox，再投递到 **Redis Stream** 作为网关消费与补发通道。Redis Stream 承担的是“可回放的投递日志”，而不是可以跳过真源的唯一事实来源。
* **WebSocket 出口层**：只在网关最后一跳做“要不要发给这个会话”的判定，不把会话活性上卷污染到任务生产侧。

> **【架构决策权衡：为什么不在写入 Redis 前遏制进行态】**
> 进行态是否需要发送，取决于“某个具体会话是否仍然活跃且仍然订阅”。这天然是一个 **Session 级** 事实，而不是 Task 级或 User 级事实。在多实例部署下，同一 `userId` 可能同时拥有多个在线设备、多个浏览器标签页、多个不同网关节点上的连接；一个会话挂起，不等于其余会话也应停止接收。
> 因此，如果在“写入 Redis 之前”就试图根据用户活性截流，会把生产侧与网关拓扑强行耦合，并制造灾难性误伤：某一台机器上观察到的“无人活跃”，可能只是一块局部真相。我们最终拍板：**进行态的遏制权只属于 WebSocket 出口层，不属于任务生产层。**

### 8.4 多实例网关下的活性门控与扇出策略
在 WebSocket 网关多实例部署前提下，进行态通知采用 **A 方案**：
* **只向显式订阅了该任务或任务列表的 `ACTIVE` 会话扇出进行态**。
* 不因为同用户另一个会话在线，就把进行态广播给该用户的全部会话。
* 不选举“用户级唯一 leader 会话”承担进行态；每个会话独立依据其订阅关系与活性状态决定是否接收。

这样定义的好处是：
* 避免同一用户多设备场景下，把无关页面或后台标签页淹没在高频 `PROCESSING` 噪声中。
* 保持订阅语义单纯：谁订阅、谁活跃、谁接收。
* 让网关扩容时不必共享庞大的“用户级正在观看什么”全局瞬时状态，只需聚焦于会话本地事实。

### 8.5 活性信号的来源、传递与状态机
会话活性信号只由 **WebSocket 网关** 根据 transport heartbeat 与 application heartbeat 推导，不由任务生产者推导。推荐将会话活性定义为三态：
* `ACTIVE`：最近仍有应用层或传输层生命迹象，可接收进行态与终态。
* `SUSPENDED`：连接尚未物理断开，但应用层心跳已超时；该会话停止接收进行态，只保留终态、ACK、Replay 与必要心跳。
* `DEAD`：连接已断开或 transport timeout，停止一切在线投递。

信号传递规则如下：
* **网关本地内存态是判定真源**，用于决定这一条消息是否真正下发到某个 WebSocketSession。
* **Redis Presence 镜像仅作为跨实例可见性的 Hint**，可采用短 TTL Key / Hash 记录 `sessionId`、`userId`、`serverId`、`streamKey`、`state`、`lastApplicationAt`、`lastTransportPongAt` 等字段。
* 该镜像允许丢失、过期与重建；它只服务于运维观测、横向排障和未来可能的“无活跃订阅者时降频”优化，**不得充当 correctness gate**。

### 8.6 终态补偿与客户端 ACK 边界
终态事件的可靠性边界必须继续保持为“消费确认”与“客户端确认”两层：
* **Redis Stream Consumer ACK** 仅表示某个网关实例已经成功消费并处理了一条终态事件。
* **客户端 ACK** 才表示前端已经真正接收并确认了该终态事件，可推进用户维度的回放游标（如 `lastAckedTerminalEventId`）。

> **【架构决策权衡：为什么 Redis Stream 的消费确认不能替代客户端 ACK】**
> 如果把 Redis Stream 的 `XACK` 误当成“用户已经收到”，那么网关实例只要在写 WebSocket 前后任何窗口崩溃，都可能导致消息既无法回放、又被错误视作“已送达”。
> 因此，**网关消费确认只对网关内部负责任；用户送达确认必须继续保留独立的客户端 ACK 语义**。这也是“终态不可丢失”能够真正成立的最后一道边界。

### 8.7 运行态恢复真源去本地化：废止 `runtime_state.db`
随着 `Java` 控制面与 `Python` 计算面向多实例演进，任务目录内的 `intermediates/rt/runtime_state.db` 已不再适合作为 authoritative 恢复真源。它最大的问题并不是 SQLite 技术本身，而是**恢复事实被单机磁盘绑定**：只要原 `Python` 宿主损坏、磁盘不可达或实例迁移，最细粒度的 `llm_call / chunk / projection` 提交事实就会一起失联。

因此，本设计拍板：
* **废止 task-local `runtime_state.db` 作为恢复真源。**
* **MySQL 成为唯一的运行态恢复真源（authoritative truth）。**
* **对象存储承接不可变正文内容**，例如 `llm_record_content / chunk_record_content / projection payload`。
* 任务目录与本地文件侧只保留最终产物、审计镜像与必要的兼容读入口，不再承担恢复决策真相。

> **【架构决策权衡：为什么不保留 `SQLite + MySQL` 双真源】**
> 双真源会制造一种最坏的复杂度：Java 侧以为自己已经拿到了恢复入口，Python 侧却仍然掌握一部分只存在本地盘的 committed facts。这样一来，系统既无法真正做到跨实例接管，又会持续陷入“到底该信 MySQL 还是信本地 SQLite”的排障泥潭。
> 我们最终选择更严格的路线：**宁可一次性把恢复真相收口到中心库，也不接受长期并存两套恢复边界。**

### 8.8 存储职责划分：MySQL 全量恢复索引，对象存储保存不可变正文
当前 `runtime_state.db` 的逻辑模型整体迁入 `MySQL`，不做“只迁一半”的折衷。迁移对象包括：
* `task_meta`
* `stage_snapshots`
* `scope_nodes`
* `scope_edges`
* `llm_records`
* `llm_record_content`（索引 + locator）
* `chunk_records`
* `chunk_record_content`（索引 + locator）
* `scope_hint_plan`
* `scope_hint_latest`

其中职责明确分层：
* **MySQL**
  * 保存所有运行态恢复索引、状态机字段、依赖图、恢复 hint、attempt 边界与 blob locator。
  * 决定“什么已经 committed”“现在从哪个阶段恢复”“这条正文是否可复用”。
* **对象存储**
  * 只保存不可变、只追加、只读取的大正文内容。
  * 不承担状态机、不承担事务真相、不承担恢复入口判断。

为了避免多次重跑互相污染，所有运行态恢复对象的归属边界至少是：
* **`task_id`**：业务任务身份。
* **`attempt_no`**：该任务的第几次执行/恢复尝试。

> **【架构决策权衡：为什么 `task_id` 不足以做唯一恢复边界】**
> 同一个任务在失败重试、人工修复、Python 节点切换、全量重跑后，可能产生多轮恢复上下文。若仅以 `task_id` 作为恢复事实唯一键，旧失败 attempt 的 `llm/chunk` 与新 attempt 的运行态会互相串扰，TTL 清理也会误删新数据。
> 因此我们要求：**运行态恢复包的最小隔离边界必须是 `(task_id, attempt_no)`，而不是单独的 `task_id`。**

### 8.9 对象存储抽象与 Key 设计
对象存储采用 **S3 兼容抽象**：
* **本地开发**：统一使用 `MinIO`。
* **线上部署**：保持 S3 兼容接口，可切换 `AWS S3 / 阿里云 OSS / 腾讯云 COS / 华为云 OBS`。
* 业务代码不直接绑定厂商 SDK 语义，只暴露最小能力集：`put / get / head / delete / list_prefix`。

对象 Key 初期按 **任务 + attempt 作用域**组织，而不直接上全局内容寻址池，目的是先保证恢复包可以“整包读取、整包清理”。推荐 key 形态如下：

```text
rt/task/{task_id}/attempt/{attempt_no}/{stage}/{asset_type}/{logical_key}/{sha256}.json.zst
```

字段含义：
* `stage`：如 `stage1 / phase2a / phase2b`
* `asset_type`：如 `llm_call / chunk / projection`
* `logical_key`：如 `llm_call_id / chunk_id / projection_name`
* `sha256`：正文 canonical bytes 的内容指纹

该设计意味着：
* 相同任务 attempt 内部可做内容一致性校验。
* 清理时可直接按 prefix 删除整个恢复包。
* 全局跨任务去重暂不在第一阶段引入，避免把恢复包 TTL、引用计数与垃圾回收复杂度耦合到同一轮改造里。

### 8.10 同步提交协议：先写对象存储，再封口 MySQL
对象正文采用 **同步提交（Synchronous Commit）** 语义：
1. 先把正文序列化为 canonical bytes，必要时压缩（如 `zstd`），并计算 `sha256` 与 `size_bytes`。
2. **先同步写入对象存储**。
3. 对象写成功后，开启 `MySQL` 事务，原子写入：
   * `llm_records / chunk_records / projection` 的 committed 元数据；
   * 对应的 content locator（`blob_key / content_hash / size_bytes / content_encoding`）；
   * `stage_snapshots / scope_nodes / scope_edges / scope_hint_*` 等恢复索引；
   * 当前阶段 checkpoint 与 retry directive。
4. **只有 MySQL 事务提交成功，才视为这次 committed fact 真正成立。**

失败语义明确如下：
* **对象存储失败**：本次提交直接失败，不写 MySQL committed 索引，不落本地临时真源，调用方按幂等语义重试。
* **对象存储成功但 MySQL 事务失败**：允许产生少量 orphan blob；后台 sweep 任务按 `blob_key` 前缀与 `content_hash` 对账后清理。

> **【架构决策权衡：为什么选择“先写对象存储，再封口 MySQL”】**
> 反过来如果先把 locator 写进 MySQL，再去上传 blob，会制造一种更危险的不一致：数据库已经宣布“这份内容可恢复”，但对象正文实际上并不存在。相比之下，允许少量 orphan blob 只是存储泄漏问题，后台可以对账回收；而“索引存在、正文缺失”会直接破坏恢复正确性。
> 因此，本设计坚持：**MySQL 负责封口 committed truth；对象存储负责提供正文素材，但永远不单独宣布提交成功。**

### 8.11 恢复包生命周期：仅失败/中断任务保留 72 小时
运行态恢复对象统一视作 **Recovery Package**。它不是长期业务资产，而是为了失败恢复、人工修复与延迟补偿而保留的短生命操作性资产。

生命周期策略如下：
* **运行中任务**：完整保留当前 attempt 的恢复包。
* **正常完成任务**：在最终产物 durable 落地、发布完成并完成必要审计后，**立即清理**运行态恢复包；不进入 72 小时保留窗口。
* **失败 / 中断 / `MANUAL_RETRY_REQUIRED` 任务**：完整保留恢复包 **72 小时**。
* **超过 72 小时**：恢复包整体过期并被整包清理，任务进入 `FULL_RERUN_REQUIRED` 语义。

恢复包的保留和清理坚持“重恢复资产”和“轻量恢复索引”同生共死：
* **同 TTL 保留**
  * `llm_record_content`
  * `chunk_record_content`
  * `projection payload`
  * `stage_snapshots`
  * `scope_nodes / scope_edges`
  * `scope_hint_*`
* **过期时整包删除**
  * `MySQL` 中的 runtime rows
  * 对象存储中 `rt/task/{task_id}/attempt/{attempt_no}/...` 前缀下的正文 blob

> **【架构决策权衡：为什么不让轻量恢复索引长于重恢复资产】**
> 这些索引的价值，本质上是帮助系统找到可复用的 committed 正文。一旦 `llm/chunk/projection` 等重恢复资产已经过期清理，仅剩索引并不能继续支撑“从最小恢复单元续跑”，系统最终仍只能全量重跑。
> 因此我们明确：**恢复索引不是长期产品资产，而是恢复包的一部分。** 真正需要长留的只应是最小审计摘要，而不是恢复索引本身。

### 8.12 前端身份模型、终态 ACK 与重连收敛顺序
前端实时链路的身份边界至少拆成三层：
* **`user_id`**：业务归属身份。
* **`device_id`**：稳定设备 / 安装身份。
* **`transport_session_id`**：一次 WebSocket 连接身份。

在“同设备多标签页不共享 ACK cursor”的要求下，仅靠一次连接级 `transport_session_id` 不足以支撑断线重连后的游标延续。原因是连接一旦重建，`transport_session_id` 就天然变化，旧 ACK cursor 会一起失效。因此推荐引入：
* **`page_session_id`**：单个标签页 / 窗口稳定持有的页面会话身份，仅服务于终态 ACK/replay。

推荐的 ACK cursor 主键为：
* **`user_id + device_id + page_session_id`**

而不是直接用瞬时的 `transport_session_id`。若协议层仍坚持“ACK 按 session”，则必须先明确：该处的 `session` 指的是 **稳定页面会话**，而不是一次性传输连接。

终态删除与重放语义如下：
* 同一设备的不同标签页 / 窗口各自维护独立 ACK cursor，不共享。
* 终态事件在**所有未过期的 `page_session_id` 均已确认**后，才允许提前删除。
* 若超过终态 replay 保留时间，则不再逐条补发历史事件；新连接直接走 `full snapshot` / `changes` 重建当前状态。

客户端重连后的收敛顺序固定为：
1. 使用 `user_id + device_id + page_session_id + lastAckedTerminalEventId` 建立连接；
2. **先 replay 终态事件**；
3. 再拉取或对账当前快照（`snapshot / changes`）；
4. 最后恢复任务级 / 列表级订阅。

设备离线语义也必须拆开：
* **短时离线**：只影响实时投递，不影响任务执行；恢复后通过 replay + snapshot 收敛。
* **永久失联**：只影响会话与 cursor 的保留，不影响任务生命周期；过期 `page_session_id` 不再阻塞终态事件删除。

### 8.13 Lease 续期、软硬心跳与 Fencing Token
本设计中的 lease 不只是“租约过期时间”，而是一套带身份校验的执行权协议。

角色边界如下：
* **Java orchestrator**：唯一的 lease 续期者。
* **Python worker**：不直接续租；仅通过 `soft heartbeat + hard heartbeat` 向 Java 证明“执行链仍然活着且仍在推进”。

推荐的续租前提全部满足时，Java 才允许续租：
* Java 仍能访问 `MySQL`；
* Java 仍持有当前任务 attempt 的最新 lease 身份；
* `soft heartbeat` 新鲜，证明执行链未失联；
* `hard heartbeat` 没有超过该阶段允许的最大“无推进窗口”。

一旦任一条件失效：
* Java **立即停止续租**；
* Java **立即 self-fencing**，不再宣布任何新的 committed fact；
* 剩余接管由 lease 自然过期后交给 sweep 处理。

为避免旧 owner 在网络抖动、GC 停顿或滞后重试后继续写入真相库，本设计要求：
* 每次 `claim` 成功时都轮转新的 **`task_lease_token`**；
* 所有 committed 写都必须携带当前 token 校验；
* token 不匹配时，写入必须被拒绝并记录审计日志。

如后续需要把长阶段进一步切细，可在 `task_lease_token` 之上再引入：
* **`stage_execution_token`**：`task_id + attempt_no + stage` 粒度的细化写权限标识。

> **【架构决策权衡：为什么 lease 必须和 fencing token 成对出现】**
> lease 只能回答“此刻理论上谁应该拥有执行权”，却不能阻止旧 worker 在租约失效后继续延迟写入。`fencing token` 的职责正是把“理论 owner”变成“唯一允许写 committed truth 的 owner”。没有 token，sweep 接管后老 worker 仍可能继续写，系统会重回 split-brain。

### 8.14 RabbitMQ、Sweep 与 DLQ 的职责边界
三者的职责必须彻底拆开：
* **RabbitMQ**：低延迟唤醒总线（wakeup bus），不是业务真相库。
* **Sweep**：基于 `MySQL` 真相库的补偿纠偏器，不扫描 MQ，不把 MQ 当真相。
* **DLQ（死信队列）**：只收“消息封装级 poison message”，不承担业务任务恢复。

RabbitMQ 的正确语义是：
* consumer 在 **`claim` 成功后立刻 ACK** wakeup message；
* 不等待“整个任务真正跑完”才 ACK；
* wakeup envelope 只表示“有人来检查这条任务是否可执行”，不表示“任务已经完成”。

因此，`lease` 过期的运行中任务不应期待 RabbitMQ 自动重新处理。因为：
* 原 wakeup 很可能早已在 `claim` 成功后被 ACK；
* 业务执行中的崩溃、假存活、超时与接管窗口，RabbitMQ 并不知道；
* 这些对象只能由 **基于 DB 真相的 sweep** 重新发现。

DLQ 只处理以下场景：
* 消息反序列化失败；
* envelope 字段非法；
* 消费器处理 wakeup envelope 时持续抛出不可恢复异常。

DLQ **不处理**：
* lease 过期；
* 业务执行失败；
* 运行中节点崩溃后的接管；
* 恢复包过期或损坏。

### 8.15 生产级 Sweep 模型：按对象分拆，而不是一个“大扫把”
更生产级的 sweep 不是一个高频全表扫描线程，而是多类对象、多条补偿链路：

* **`Execution Lease Object`**
  * 关注 `status / lease_owner / lease_token / lease_expires_at / next_retry_at`
  * 职责：重新发现可接管任务
* **`Dispatch Redrive Object`**
  * 关注 `last_wakeup_requested_at / last_claimed_at / dispatch_generation / dispatch_state`
  * 职责：判断 wakeup 是否真正丢失，而不是把正常 backlog 误判成漏唤醒
* **`Recovery Blob Object`**
  * 关注 `blob_key / content_hash / expires_at / package_status`
  * 职责：清 orphan blob、过期前缀和损坏正文
* **`Session Cursor Object`**
  * 关注 `user_id / device_id / page_session_id / last_seen_at / last_acked_terminal_event_id`
  * 职责：过期页面会话、终态 ACK cursor 清理

本设计**不再单列“卡死 sweep 对象”**。原因是：
* 只要 Java 的续租已经严格绑定 `soft/hard heartbeat + MySQL 可达 + token 有效`，
* 那么“Python 死锁但 Java 仍盲续 lease”的情况本应由 owner 自己停止续租来暴露，
* 后续只需要 sweep 处理“lease 已过期，可被重新接管”的对象即可。

#### 8.15.1 Dispatch Redrive 如何避免误伤 MQ backlog
对于长任务系统，“消息还在 MQ 里排队、只是暂时没有空闲 consumer”并不等于漏唤醒。因此：
* `Dispatch Redrive` 不应只按 `status age` 判断；
* 必须结合 `dispatch_state` 与 `last_wakeup_requested_at`；
* 只有在 **超过 redrive timeout 且仍无 claim** 时，才允许重发 wakeup。

推荐基线：
* `dispatch_redrive_timeout` 不应使用过短固定值；
* 它至少应大于当前系统 **P99 enqueue-to-claim latency** 与队列积压窗口；
* 初始保守值建议 `120s ~ 300s`，后续由运行指标回调。

#### 8.15.2 Expired Lease Reclaim 与建议周期
`Execution Lease Sweep` 主要处理两类对象：
* `status = RUNNING` 且 `lease_expires_at < now`
* `status = RETRY_WAITING` 且 `next_retry_at <= now`

推荐的初始节奏：
* `lease_duration = 30s`
* `renew_interval = 10s`
* `expired_lease_reclaim_sweep = 5s ~ 10s`
* `dispatch_redrive_sweep = 30s`
* `recovery_blob_sweep = 15m`
* `session_cursor_sweep = 5m`

这些值不是业务真理，而是生产起步值；后续应根据：
* `claim_conflict`
* `expired_lease_recovered_count`
* `p99 enqueue-to-claim latency`
* `queue backlog`
* `fencing_reject_count`
进行回调。

#### 8.15.3 更像大厂的工程约束
生产级 sweep 至少需要以下纪律：
* **统一 ClaimService**
  * MQ consumer 和 sweep 必须调用同一套 `claim` 逻辑，不允许各写一套 SQL。
* **分片扫描**
  * 按 `task_id` 哈希或主键区间分片，避免全表硬扫。
* **小批量 + 游标推进**
  * 每批固定上限，防止 sweep 反向拖垮真相库。
* **审计与指标**
  * 必须记录 `scan / claim / redrive / reclaim / orphan cleanup / fencing reject`。
* **指数退避**
  * 坏任务不能被 sweep 高频反复捞起，必须受 `retry_count / next_retry_at / MANUAL_RETRY_REQUIRED` 约束。

### 8.16 CAP / BASE 的分层定位
本系统不应被描述为“整体纯 BASE”或“整体纯 CP”，更准确的结论是：
* **真相层（Truth Layer）偏 CP**
* **边缘链路（Edges）偏 BASE**

具体表现为：
* **Accept Plane**
  * `MySQL durable accept` 是真相，偏 CP；
  * `202 / 排队 / 退避 / 削峰` 是对可用性的 BASE 式退化。
* **Execution Plane**
  * `MySQL lease + fencing token + recovery index` 是 CP 真相；
  * `RabbitMQ wakeup + retry + sweep compensation` 是 BASE 式编排。
* **Publish Plane**
  * `published pointer` 的切换必须是 CP；
  * CDC、缓存刷新、外围副本传播采用最终一致。
* **Realtime Plane**
  * 进行态快照流、presence、Pub/Sub 明显偏 BASE；
  * 终态则是“CP-backed truth + BASE delivery”，先有 durable truth，再有 replay/ack。

因此，本设计的总体定位应表述为：
> **CP truth + BASE edges**
>
> 系统不试图在全局上“同时拥有 CAP 的全部好处”，而是先承认分区一定存在，再按边界选择：对真相对象优先保一致性，对加速、通知和补偿链路优先保基本可用与最终一致。

## 9. 资源级安全权限保护网
本设计摒弃了“单用户删除了该记录，系统就把它抛掉”那种单纯 CRUD 错觉。在此，公域底层材料被剥夺了 C 端客户层级的物理删除权。用户的每一次 “删除/撤销”，本质上都仅是在对 `user_job_binding` 触发 `Unsubscribe`（退订）或 `Hide`（屏蔽视线）。真身资源在后台毫发无损，甚至仍在不遗余力为别的人持续推流贡献性能红利。

## 10. 高洪峰下的软性限截与动态护航网
不能让“接更多单”演变成毁灭全局执行流处理的导火索。背压过载保护通过“动态软分数判定模式（Scoring Metrics Model）”决定入口策略。

### 10.1 模式（Mode）切面与分数卡口制
为防范由单一系统毛刺导致防卫策略来回抽帧（Flapping），评判基准囊括包含数据库活跃连接度、延迟时位、单机线程队列壅塞度等多维参数综合判定：
* `NORMAL` (<2分)：全速巡航。
* `BUSY` (2~4分)：切除乐观通道，谨慎复用。
* `FLOOD_PROTECT` (>=4分，最高硬限极值为 6 截断)：放弃同步抵入等待计算。全部降维抛弃接听直抵 `202 ACCEPTED` （或者超额阈值时 `429/503`）。

> **【架构决策权衡：为何不迷信固定 QPS 漏桶拦截】**
> 我们使用动态软分打卡截断（而非限制每秒必须仅进 5000 次死限）。因为入口的真正承载威胁永远是“系统内下游还能否呼吸”。一个短时轻量级的 20000 冲峰在缓存充裕与计算池闲置时应被完美吸纳；而在数据库阻塞发生时，即便只漏进 50个 QPS 也会引起连环死锁反应。以“系统指标反应池水位（JVM/DB连接余缺等真实负载压强）”而非外网指标来定界拦截，才具有弹性防护力与大将之风。

## 12. 客户端崩溃态的强恢复契约要求
（重点指引研发人员）客户端重启恢复不依赖于绝不能丢失或必须固化的 `submission_token`。那不现实也不具备强可用性。
* 系统给予的强兜底是：只要你在发起 HTTP POST 实体之前在客户端存储（SQLite / IndexedDB / LocalStorage）中把生成的那个代表心智意向的 `Idempotency-Key` 存好了！
* 那么即便你拔掉网线没收到服务端的 `202 / 200`，重连后只要沿用该 Key。服务端利用它做真源或二级重返的检索回捞，势必能重新把你在彼岸尚未跑完或已早定局的生命进度归还原状。

## 13. 系统仍待精修打磨的空隙
*  由于分离出了全局真库，底层 `MySQL` 入口事务锁行数范围。
*  基于 `attempt` 衍生后的更多 Provider 阶梯分型，继续穷举映射网络上的更多不可预估灾难，补充进我们的 Degradation Class 异常库之中。

## 14. 结语
本入口体系是《异构长文本/极速音视频分析处理平台》抗击第一波峰值海啸的最前沿大坝。在经历本次彻底剖解重构后，系统明确了自己：
**何时该妥协以柔克刚保生存（AP 模式 / BASE 软状态），何时必须向死而生维护事实（CP 唯一性断言 / DB 底座锁）。**
这并不只是一份功能表，而是一份为了“即使系统崩坏依然知道能从哪里重建秩序”而签下的架构契约书。
