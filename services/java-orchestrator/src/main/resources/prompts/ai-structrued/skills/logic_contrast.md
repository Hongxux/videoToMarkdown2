# Skill: 对比/转折关系精修（logic_contrast）

当 section 中存在"A vs B"、"虽然A但B"、"旧方案 vs 新方案"等张力对比结构时，适用本 skill。

---

## 精修规则

### 1. 对比双方结构对称
- 对比的两方必须**维度一一对应**：如果 A 方从"性能"、"成本"两个维度描述，B 方也必须从同样的维度描述
- 禁止出现"A 说了三个方面、B 只提了一个"的不对称结构
- 如果原文只有 A 方的信息，不得为 B 方编造数据，**直接省略该维度的对比**，只呈现原文提供的一方

### 2. 对比维度显性化
- 必须在父节点或首行**显式标注对比的维度/视角**，且维度必须来自原文已有的论述角度
- 例如：不要写 `**A 和 B 的区别**`，应写 `**A 与 B 的核心差异（从{原文涉及的维度}对比）**`

### 3. 对仗句式呈现张力
- 对比双方的标签应使用**成对的反义/互补词汇**，增强语义张力
- 例如：
  - `**旧版（同步阻塞）的瓶颈**` vs `**新版（异步非阻塞）的破局**`
  - `**传统方案的代价**` vs `**改进方案的红利**`

### 4. 结论性总结必备
- 对比结构的末尾必须有一个**结论节点**，显式指出"关键差异点在哪"或"为什么选 B 而非 A"
- 结论必须从原文内容中归纳得出，**禁止添加原文未提及的新论点**

### 5. 忠实原意红线
- 对比结构的每一项内容必须**严格来源于原文**
- 禁止为了凑结构对称而编造原文没有的数据、百分比、具体数值、案例或论据
- 如果原文只说了"性能更好"，精修后也只能说"性能更好"，不得擅自补充"提升数倍"之类的量化描述

### 6. 递归深拆
当对比项内部仍然包含复句或多层含义时，必须继续递归拆解为更深的子节点——通过改写重组保证每个最终叶子节点都是一个**单义短句**。

### 正向示例

**原文输入：**
> 传统线程模型会阻塞物理线程，而虚拟线程不会阻塞物理线程，性能更好。

**❌ 错误精修（添加了原文没有的内容）：**
```markdown
- **线程模型对比**：
    - **传统模型的瓶颈**：
        - 物理线程在 I/O 等待期间被操作系统挂起，无法复用。
        - 线程数受限于系统资源，高并发场景下成为硬件级瓶颈。
    - **虚拟线程模型的破局**：
        - 请求吞吐量较传统方案提升数倍。  ← ❌ 原文没说"数倍"
        - 极少的物理线程即可支撑海量并发任务。 ← ❌ 原文没说"海量"
```

**✅ 正确精修（仅重组原文已有内容）：**
```markdown
- **线程模型对比（从阻塞行为角度对比）**：
    - **传统模型（同步阻塞）的瓶颈**：
        - **阻塞行为**：会阻塞物理线程。
    - **虚拟线程模型（非阻塞）的破局**：
        - **阻塞行为**：不会阻塞物理线程。
    - **对比结论**：虚拟线程避免了物理线程被阻塞，因此性能更好。
```
