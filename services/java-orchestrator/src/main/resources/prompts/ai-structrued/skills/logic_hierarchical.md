# Skill: 分层关系精修（logic_hierarchical）

当 section 中存在明显的上位概念与下位细节的包含/层级关系时，适用本 skill。

---

## 精修规则

### 1. MECE 互不重叠原则
同一层级的子节点之间必须满足 **MECE（Mutually Exclusive, Collectively Exhaustive）**：
- **互不重叠**：任意两个同级子节点的语义边界不得交叉。如果发现两个子项说的是同一件事的不同侧面，必须合并或重新划分边界。
- **完整覆盖**：所有同级子节点合在一起应能完整覆盖父节点的语义范围。如果原文确实没有涉及某个维度，**直接不写**，不得编造内容。

### 2. 分层维度显性化
- 每一层缩进层级的父节点必须**显式说明分层的维度/视角**
- 例如：不要简单写 `**方面一**`，而应该写 `**技术层面（基础设施视角）**`
- 分层维度来源必须可追溯到原文，不得凭空发明分类体系

### 3. 层级深度检查
- 检查是否存在**虚假层级**：某些缩进层级只有一个子节点，说明这个层级不应存在，应该合并到父节点
- 检查是否存在**层级过浅**：某个子节点内包含了多个本应再次拆分的概念，需要进一步下钻

### 4. 忠实原意红线
- 分层归纳的所有内容必须**严格来源于原文**
- 分类标签只能从原文的论述中归纳提炼，禁止凭空发明原文未涉及的分类维度
- 禁止为了凑 MECE 而编造原文没有提到的层级、数据或百分比

### 5. 递归深拆
当某个层级节点内部仍然包含复句或多层含义时，必须继续递归拆解为更深的子节点——通过改写重组保证每个最终叶子节点都是一个**单义短句**。

### 正向示例

**原文输入：**
> 系统优化主要体现在两方面：内存管理优化后占用降低了，响应速度也加快了。

**❌ 错误精修（编造原文没有的分类和数据）：**
```markdown
- **系统优化维度拆解（按资源类型互斥划分）**：
    - **计算资源优化**：响应速度加快，CPU 利用率更均匀。 ← ❌ 原文没说 CPU 利用率
    - **存储资源优化**：内存占用降低约 30%，GC 停顿减少。 ← ❌ 原文没说 30% 和 GC
    - **网络资源优化**：连接复用率提升，带宽浪费减少。 ← ❌ 原文完全没提网络
```

**✅ 正确精修（仅归纳原文已有内容）：**
```markdown
- **系统优化（两个方面）**：
    - **内存管理优化**：优化后内存占用降低。
    - **响应速度优化**：响应速度加快。
```
