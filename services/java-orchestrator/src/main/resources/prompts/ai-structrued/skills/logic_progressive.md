# Skill: 递进关系精修（logic_progressive）

当 section 中存在层层递深、由浅入深、从表象到本质的升级关系时，适用本 skill。

---

## 精修规则

### 1. 递进点在语义标签中显性化
- 每个递进层级的语义标签必须**体现递进的梯度**，让读者仅通过标签就能感知由浅入深的层次跃迁
- 禁止使用"方面一"、"方面二"这类无递进感的平铺标签

**递进梯度参考维度**：
- **深度递进**：表象 → 机制 → 本质（如：`现象` → `底层原因` → `根本原理`）
- **范围递进**：局部 → 全局 → 超越（如：`个人影响` → `团队影响` → `行业影响`）
- **时间递进**：短期 → 中期 → 长期（如：`即时止血` → `中线巩固` → `长线筑基`）
- **抽象递进**：具象 → 抽象 → 哲理（如：`操作层面` → `策略层面` → `认知层面`）

### 2. 递进连接词/标注
- 在递进层级的标签中，使用能体现递进方向的词汇：`进一步`、`更深层`、`不仅如此`、`由此延伸`等
- 或在父节点标注递进方向：`**（由浅入深递进展开）**`

### 3. 忠实原意红线
- 递进层级的每一级内容必须**严格来源于原文**
- 递进梯度的标签命名只能从原文的论述中归纳，禁止编造原文没有的递进层次
- 如果原文的递进链条存在跳跃（从浅到深中间缺失某层），**直接省略缺失的层次**，不得替原文补全

### 4. 递归深拆
当某个递进层级内部仍然包含复句或多层含义时，必须继续递归拆解为更深的子节点——通过改写重组保证每个最终叶子节点都是一个**单义短句**。

### 正向示例

**原文输入：**
> 这次改造立竿见影，不仅止住了宕机的血，还奠定了后来五年的架构底座。

**❌ 错误精修（编造原文没有的层次）：**
```markdown
- **改造收益（按影响时间线由近及远递进）**：
    - **浅层收益**：改造立竿见影。
    - **中层收益（中线架构筑基）**：形成了可复用的技术资产沉淀。 ← ❌ 原文没说"技术资产沉淀"
    - **深层收益（长线业务杠杆）**：成功容纳了后续千万级并发洪峰。 ← ❌ 原文没说"千万级"
```

**✅ 正确精修（仅重组原文已有内容）：**
```markdown
- **改造收益（由近及远递进）**：
    - **即时效果（止血）**：改造立竿见影，止住了宕机。
    - **长远效果（筑基）**：奠定了后来五年的架构底座。
```
