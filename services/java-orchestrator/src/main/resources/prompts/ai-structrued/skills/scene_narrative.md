# Skill: 叙事复盘 / 故事经历场景（scene_narrative）

当 section 涉及时间线、个人经历、故事回顾、阶段性复盘时，适用本 skill。

---

## 精修规则

### 1. 宽容粒度 — 保持叙事连贯
- 按"**阶段 / 关键节点 / 里程碑**"进行结构组合
- 允许在一个列表项中保留连绵起伏的几句话段落，**不要将叙事文本过度碎片化**
- 叙事文本的情境感和情感张力优先于结构工整

### 2. 显性化故事弧线
- 用标签点明故事流转过程中的**角色/阶段**，推荐标签包括但不限于：
  - `**背景**` / `**导火索**` / `**转折点**` / `**高潮行动**` / `**结果**` / `**沉淀反思**`
- 标签应根据具体故事内容灵活选择，不要机械照搬所有标签

### 3. 时间锚点保留
- 原文中的时间节点（如"2012年"、"大促期间"、"一个月内"）必须保留并在标签中体现
- 例如：`**2012 年大促 — 灾难性背景**` 而不是泛化为 `**背景**`

### 正向示例

**输入：**
> 2012年的时候，我们的整个数据库架构还是单体 MySQL，当时大促期间业务量涨得飞快，每天订单量破十万，导致主库经常在晚高峰被打死，CPU直接飙到100%。大家当时急坏了，连续通宵了三天。最后我们决定破釜沉舟，在一个月内强行推行了分库分表策略，并且引入了 Redis 做读写分离，这次改造不仅让我们团队活下来了，还顺势奠定了后来五年的高并发可用架构底座。真的是险招。

**期望精修输出：**
- **早期的灾难性背景（单体时代的致命隐患爆发）**：当时正值 2012 年首个大促引爆的初春，赖以生存的**[[单体架构 MySQL]]**陷入苦痛绝境。
    - **流量环境极限激增**：
        - **诱因**：全站大促期间的巨额流量疯狂涌入。
        - **量级**：系统日均订单量直破十万大关。
    - **极端的系统雪崩表征**：
        - **高危时间点**：每逢晚高峰时段必瘫。
        - **核心瘫痪点**：数据库主库被巨额的双向读写流量强行击溃。
        - **崩溃指标**：==服务器 CPU 负载直接达到 100% 满点死锁==。
    - **沉重的人事灾难外延**：
        - **状态**：全组研发被迫陷入随时高强度的绝望救火状态。
        - **极致代价**：连续熬过三个大通宵苦苦支撑生命线。
- **大转折逆袭图存（生死竞速下的破冰大改造决议）**：面对令人焦头烂额的无解绝壁，项目组核心团队最终一咬牙拍板确定突围方案。
    - **决策基调**：即刻破釜沉舟，执行一套兼具凶险与激进的架构焕新突围计划。
    - **并行突围行动一（打碎木桶强推分库分表）**：
        - **动作**：大张旗鼓开启**库表水平拆分工程**。
        - **极光阴要求**：勒令团队必须在极端的"一个月以内"生死限期内完成。
    - **并行突围行动二（构建弹簧垫引入缓存网）**：
        - **动作**：极其果断极速地拉通引入了 **[[Redis]]** 高可用缓存机制。
        - **战略目的**：在系统最前线提前接管并分摊掉最为沉重的查询读流量洪灾。
- **阶段性战役赢局总结（痛感唤醒后的化蛹成蝶）**：
    - **浅层收益（短线急救止血解围）**：
        - 这次极端狂暴的改造之举立竿见影。
        - 彻底消解了动辄引发全站断线雪崩的定时宕机炸弹。
    - **深层红利（长线坚实骨架夯土）**：
        - **心态层**：从被动救火的"消防员"状态中痛极思变、逼迫觉醒。
        - **架构层**：在这场不期而遇的突围黑障战中，极其提早锻造夯实出了足以从容承接后续足足五年的宏伟架构底座。
        - **业务层**：成功容纳了后续千万级并发暴雨狂潮的冲击。
