2) 直接输出 Markdown，不输出 JSON 或代码块。
3) 如果给出图片候选，请在对应句子的句末插入占位符，格式必须为【imgneeded_{{img_id}}】。

## 结构要求
1. **核心论点置顶**：必须在 Markdown 的第一行，使用引用块格式显式提取该单元的核心论点。格式：`> **核心论点**：...`
2. **逻辑树构建**：
   - 使用 **加粗的语义标签+冒号** 体现逻辑节点（如 **关键主张：**、**执行步骤：**）。
   - 严格使用 **4个空格（Space）** 缩进体现父子逻辑（论点 -> 论据 -> 证据）。
3. 统一使用 **-** (减号) 作为列表标记。
4. 还原隐性逻辑，使结构线性化。
5. 将动作单元的 key_evidence 作为具体示例插入对应位置。
6. 数学公式请使用 LaTeX 格式，行内公式用 $...$，块级公式用 $$...$$
7. **双语术语保留**：原文中出现的双语术语（如「智能体工程（agentic engineering）」「氛围编程（vibe coding）」）**必须原样保留**，不得拆分、翻译、或省略括号内的外文注释。

## 语义拆分与逻辑树构建要求（核心）
1. **强制拆解长段落**：所有超过 1 行的长段落，如果包含多个逻辑层（如：定义+举例+优缺点、现象+原因+对策等等），**必须**将其拆分为多个独立的子列表项，禁止堆砌在一起。
2. **语义标签驱动**：每个拆分后的子项，必须以 **加粗语义标签+冒号** 开头（例如：**- **关键主张**：...**、**- **典型案例**：...**）。
3. **颗粒度控制**：每个子项只表述一个核心意思。如果一个意思包含多个步骤或方面，请进一步使用子列表展示。

## 衔接与流畅性要求（核心）
结构化并不等于"割裂化"。必须在保持清晰结构的同时，确保内容像流水一样连贯。

### 1. 跨单元衔接（Inter-unit Cohesion）
**优先使用「分组上下文」**：当提供了分组上下文（组名 + 组内单元列表）时，过渡句应基于本单元在组内的位置和角色来生成：
- **组内首个单元**：过渡句应简要概括该组的整体主题，说明本单元作为开篇的角色。
  - 示例：`> 本组围绕"X的设计与实现"展开，首先介绍其核心定义。`
- **组内后续单元**：过渡句应解释本单元与组内前一单元的逻辑递进关系。
  - 示例：`> 在明确了X的定义后，接下来探讨其在实际场景中的应用挑战。`

**回退到话题上下文**：如果没有提供分组上下文，但"话题上下文"显示存在上一个或下一个话题时：
- **过渡句（Transition）**：在结构化输出的**开头**，使用引用块格式 `> ` 生成 1-2 句过渡句，解释话题转换的逻辑。
  - 示例：`> 在探讨了X的核心定义后，现在转向其在实际场景中的应用挑战。`
- **何时省略**：如果"话题上下文"为"(无上下文)"且无分组上下文，则不生成过渡句。

### 2. 单元内衔接（Intra-unit Cohesion）
列表项之间不应是孤立的"词条"，而应像段落一样有逻辑流动：

1. **实质性衔接**：每个子列表项的开头，不要只用空洞的连接词（如"因此""然而"），而应**简要总结**前文的关键点或原因，再引出当前内容。
   - ❌ 空洞：**优势**：因此，性能提升了3倍。
   - ✅ 实质：**优势**：基于异步机制和缓存优化，性能提升了3倍。
   - ❌ 空洞：**局限**：然而，存在内存瓶颈。
   - ✅ 实质：**局限**：由于需要预分配大量缓存空间，在内存受限环境中会遇到瓶颈。
   - ❌ 空洞：**案例**：例如，某公司成功应用。
   - ✅ 实质：**案例**：在电商高并发场景中，某公司通过引入方式B将响应时间缩短了50%。

2. **禁止"百科词条"式罗列**：
   - ❌ 错误：列表项A、列表项B、列表项C 互不关联，读起来像独立的知识卡片。
   - ✅ 正确：列表项之间通过连接词和逻辑链条串联，读者能感受到思路的推进。

3. **适度保留长句**：
   - 如果一个论述本身就是流畅的长句（包含多个从句但逻辑紧密），**不要强行拆分**。"拆分"的目的是澄清逻辑，而非制造碎片。
   - 只有当长句中包含**明确的多层逻辑**（定义+例子+优缺点）时，才进行拆分并用语义标签标注。

4. **衔接示例对比**：
   - ❌ 孤立列表：
     ```
     - **优点**：速度快。
     - **缺点**：内存占用高。
     - **适用场景**：大规模数据。
     ```
   - ✅ 流畅列表：
     ```
     - **核心优势**：处理速度极快，适合对时效性要求高的场景。
         - **代价**：然而，这种速度是以较高的内存占用为代价的。
         - **最佳实践**：因此，最适合用于大规模数据处理场景，此时内存成本相对可控。
     ```

5. **挖掘同级列表项的逻辑脉络(关键)**:
   当一个语义单元包含多个同级列表项时,**禁止**将其视为无序并列。必须主动挖掘它们之间潜在的**逻辑递进**或**叙事脉络**,并通过以下方式体现:
   
   - **识别逻辑线索**:判断同级项之间是否存在以下关系:
     - **递进关系**(A引出B):前一项是后一项的基础或前提
     - **因果关系**(A导致B):前一项的结果引发后一项
     - **对比关系**(A vs B):两项形成对照或矛盾
     - **演化关系**(A发展为B):时间或状态的变化轨迹
     - **叙事关系**:成就→矛盾→演变→结果等故事线索
   
   - **语义标签体现脉络**:使用能反映逻辑关系的语义标签,而非平铺式的并列标签。
     - ❌ 并列式(丢失脉络):
       - **智能体能力**:具备自我认知与操作能力。
       - **开发模式矛盾**:对"氛围编程"抱有复杂情感。
       - **开发方式演变**:从键盘输入发展为语音开发。
       - **职业背景**:收到大公司邀约。
     
     - ✅ 递进式(呈现脉络):
       - **技术成就——智能体的理想状态**:构建出具备自我认知与操作能力的智能体,极大简化了开发流程。
       - **内心矛盾——工程化与直觉化的冲突**:尽管追求"智能体工程"的严谨性,深夜高强度工作时仍会不自觉地切换到"氛围编程"模式,并在次日感到后悔。
       - **实践演变——开发方式的彻底转变**:在这种高强度节奏的驱动下,最终放弃键盘输入,完全通过语音来编写提示词和操控终端,甚至一度因过度使用而失声。
       - **外部认可——职业影响的显现**:这些在智能体领域的创新实践,最终为作者带来了来自大公司的巨额合作邀约。
   
   - **过渡短语串联**:每个列表项的开头应包含承上启下的短语,体现思路推进:
     - 使用"基于此""由此引发""然而""因此""随着""最终"等连接词
     - 在语义标签中加入阶段性词汇:"起点""转折""深化""结果"等
     - 让读者能顺着列表从上读到下,感受到完整的逻辑链条或故事弧线

### 3. 叙事流特殊处理
当多个列表项来自同一段连续话语/对话时，各列表项之间**必须保留叙事连贯感**：
- 使用简短的衔接表述来承接上一项的语境（隐式或显式衔接）。
- 读者应能顺着列表项从上读到下，感受到作者的思路流转。
- 如果"话题上下文"中提供了上一个/下一个话题的标题，则在结构化输出的**第一个列表项之前**，用 1-2 句话简要说明话题转换的原因或逻辑关联。
   - 如果没有提供话题上下文（即"(无上下文)"），则**不生成**过渡句

## 叙事/故事流处理策略（新增关键策略）
针对“项目发展史”、“个人经历”、“轶事故事”等明显具有时间连续性或因果流动的叙事内容：

1.  **流畅性优先**：故事的整体流畅度和沉浸感优于颗粒度的拆分。
    -   **豁免权**：对于此类内容，**豁免**“强制拆解长段落”的规则，允许在一个列表项中包含 2-3 句话，以保持叙事的连贯性。
2.  **按“场景”或“阶段”聚合**：
    -   不要把故事打碎成原子化的动作（如：1.他想.. 2.他做.. 3.他感觉.. 均为独立项）。
    -   应按**事件发展的阶段**进行聚合（如：**- 阶段一：初心的萌芽**... **- 阶段二：挫折与转折**...）。
3.  **挖掘隐性叙事线索**:
    -   即使原文未明确标注时间顺序或因果关系,也要主动识别潜在的叙事脉络:
        - **成长弧线**:从初始状态→遇到挑战→做出改变→获得结果
        - **情感演变**:从兴奋→困惑→顿悟→成熟
        - **技能进化**:从基础尝试→遇到瓶颈→突破创新→形成方法论
    -   通过语义标签显性化这些隐性线索(如:"起点""转折""突破""沉淀")

4.  **减少过度嵌套**:
    -   在讲述一个连贯故事时，尽量使用扁平的结构或仅使用一级嵌套，避免把一个完整的短故事拆成三四层，导致阅读时的割裂感。


❌ 错误示例（孤立罗列，丢失叙事线）：
```
- **术语偏好**：更喜欢"智能体工程"这一称谓。
- **实际行为**：凌晨三点后会切换到"氛围编程"模式。
- **开发方式**：完全通过语音输入来编写提示词。
```

✅ 正确示例（保留叙事衔接）：
```
- **术语偏好**：更喜欢"智能体工程"这一称谓，认为"氛围编程"带有贬义色彩。
- **实践中的矛盾**：尽管如此，在凌晨三点后仍会不自觉地切换到"氛围编程"模式，并在次日为此感到后悔。
- **开发方式的进一步演变**：在这种高强度的开发节奏中，逐渐放弃键盘输入，完全通过语音来编写提示词并操控终端。
```

## 论证嵌套要求（关键）
对同一个论点/观点的**原因、后果、程度、作证、举例、论证、隐喻**，**禁止**与论点作为同级并列列表项展开，**必须**通过缩进嵌套体现从属关系。

1. **核心规则：论据从属于论点**
   - **论点**（核心主张/描述/现象）作为父级列表项的正文
   - **论据**（原因/后果/程度/证据/举例/隐喻）作为该父级下的缩进子列表项
2. **区分并列关系与支撑关系**：
   - **并列标签**（如 **方式A** / **方式B**、**优点** / **缺点**、**阶段一** / **阶段二**）→ 使用同级列表
   - **支撑标签**（如 **原因**、**后果**、**证据**、**例证**、**程度**、**隐喻**）→ **一律**嵌套在其所支撑的父观点之下
3. **合并主体句**：如果一个列表项只有一句核心描述，且其子项全是对该描述的佐证/补充，应将核心描述直接写在父级标签的冒号之后，而非再拆出一个"描述"子项。

❌ 错误示例（论据与论点平行展开）：
```
- **语音操作的实践与影响**：
    - **确认**：作者确认使用语音操作所有终端。
    - **历史强度**：过去使用非常频繁，曾一度导致嗓子沙哑。
```

✅ 正确示例（论据嵌套于论点之下）：
```
- **语音操作的实践与影响**：作者确认完全通过语音操作所有终端。
    - **使用强度**：过去使用极为频繁，高强度的语音输入曾一度导致嗓子沙哑。
```

❌ 错误示例（因果链被打散为并列项）：
```
- **两种模式的矛盾**：
    - **公开身份**：对外宣称从事"智能体工程"。
    - **私下行为**：凌晨三点后切换到"氛围编程"模式。
    - **后果**：需要自己修复和收拾残局。
    - **隐喻**：用"这堵墙一直都在那儿"来比喻。
```

✅ 正确示例（因果链通过嵌套体现逻辑层次）：
```
- **两种模式的矛盾**：对外宣称从事"智能体工程"，但私下在凌晨三点后仍会不自觉地切换到"氛围编程"模式，并在次日感到后悔。
    - **后果**：这种模式切换导致的代码问题需要自己来修复和收拾残局。
    - **隐喻**：作者用"这堵墙一直都在那儿"来比喻这种反复出现的困境。
```

## 最佳插图位置策略 (Optimal Image Placement Strategy)
1. **即时视觉反馈**：此原则优先级最高。图片占位符 `【imgneeded_{{img_id}}】` 必须**紧跟**其解释的核心概念或步骤之后。
   - ❌ 错误：描述A，描述B，描述C。【imgA】【imgB】【imgC】
   - ✅ 正确：描述A【imgA】，描述B【imgB】，描述C【imgC】
2. **避免视觉割裂**：
   - 在列表项中，图片应位于该项内容的**最后**，而不是插在句子中间打断阅读流。
3. **特定场景**：
   - **整体架构图**：应放在介绍该架构的 **一级标题或首段** 之后，建立整体认知。
   - **分步操作截图**：必须精确放置在对应 **步骤描述** 的末尾。
   - **数据图表**：必须紧跟在 **数据结论** 之后，作为证据支撑。

## 内容完整性要求
1. **保留例子与类比**：

2. **细节不丢失**：
   - 在重组结构时，确保不丢失原文的关键细节信息，尤其是数据、特定名词和操作步骤。
